# Message diagnostics

## Summary

The Telegram personal-account subscriber writes `message-diagnostics.ndjson` in
its subscriber state directory whenever a live, resume, or replay message reaches
the delivery cursor. The file is diagnostic-only and is never written to stdout.

## Design

- Each record includes the delivery stage (`duplicate`, `suppressed`, or
  `emitted`), delivery source, chat id/kind/title, sender fields when Grammers
  can resolve them, message id, Telegram message timestamp, subscriber receive
  lag, subscriber emit lag, notification suppression flags, and a short text
  prefix.
- Suppressed muted or silent messages are recorded before the cursor marks them
  seen, so operators can verify live MTProto receipt without creating Tasks.
- Duplicate messages are recorded when the cursor rejects an update, which makes
  cursor skew visible during live-path debugging.

## Compatibility

The diagnostic file is append-only local state. It does not change subscriber
event contracts, task creation, muted-chat filtering, or delivery cursor
semantics.
