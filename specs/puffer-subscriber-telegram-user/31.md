# puffer-subscriber-telegram-user update 31

The Telegram user subscriber performs one startup dialog hydration pass before
entering the raw MTProto update loop. This pass uses `Client::iter_dialogs`,
which Grammers requires to seed update state for unknown broadcast channels and
megagroups, then immediately starts `next_raw_update` for the hot path. The
pass is not a periodic polling fallback.

The same hydration pass initializes delivery suppression from the dialog
snapshot. Muted dialogs and dialogs in non-main folders are marked suppressed
before any live message is processed, and raw `UpdateNotifySettings` and
`UpdateFolderPeers` updates keep that cache current after startup. Archived
state is tracked separately from notification mute state, so an unmute update
does not accidentally unarchive a chat.

On restart, known unsuppressed chats are backfilled from the durable Puffer
delivery cursor with a bounded per-chat resume scan. First-time or newly seen
dialogs are marked seen without emitting historical backlog, and suppressed
dialogs advance the cursor without creating monitor events.
