# Telegram phone login auth restart handling

Telegram phone-number login now starts from a fresh in-memory MTProto session
instead of reusing the persisted subscriber session. This keeps imported
Telegram Desktop or macOS app auth keys out of the `auth.sendCode` path and
only overwrites the Puffer session file after Telegram accepts the code
request.

If Telegram returns `AUTH_RESTART` while requesting the code, the subscriber
discards that transient login client and retries once with another fresh
session before emitting a `login_error` for `request_code`.

Runtime re-authentication uses the same `login::start` implementation as the
initial login phase, so credential resolution, control events, pre-auth session
persistence, and restart retry behavior remain consistent.
