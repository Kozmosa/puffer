# Telegram QR 2FA Handoff

QR login no longer treats `SESSION_PASSWORD_NEEDED` from
`auth.importLoginToken` as a terminal failure. When Telegram accepts the QR
approval but requires the account's cloud password, the subscriber fetches the
password challenge with `account.getPassword`, stores the resulting
`PasswordToken` in the normal login state, emits `login_awaiting_password`,
and keeps the QR MTProto client available for `TelegramLoginSubmitPassword`.

The QR command API now returns an explicit pending, awaiting-password, or
complete outcome internally. Runtime reauth stops live streaming only when the
QR flow replaces the active client or enters the 2FA step; pending QR tokens do
not disturb the existing authorized client.
