# Telegram live active dialog poll

## Summary

The Telegram user subscriber keeps a runtime watchlist of chats that have
produced live messages and polls those chats directly during the live cursor
poll. This closes the gap where muted or archived chats can produce one live
hit but then fall out of `iter_dialogs()` ordering before subsequent messages
are delivered.

## Design

- `LiveDialogTracker` stores packed `Chat` values for recently active chats.
- Raw live messages add their chat to the tracker before delivery.
- Live cursor polling first scans tracked chats directly with
  `iter_messages(chat.pack())`, then performs the bounded recent-dialog scan.
- The tracker is in-memory only and is cleared if the Telegram client session
  is replaced during login or import.
- The watchlist is capped to avoid turning the poll loop into an unbounded
  full-account scan.

## Delivery semantics

Direct tracked-chat polling still uses the durable `DeliveryCursor`, so raw
updates, recent-dialog polling, and direct tracked polling are deduplicated by
chat id and message id. Muted or silent messages continue to be suppressed from
task emission, but they pass through delivery diagnostics and advance the
cursor.

Startup hydration no longer skips backfill for existing muted chats. When a
chat already has a cursor entry, missed muted messages are replayed through the
delivery path as suppressed diagnostics instead of silently jumping the cursor.

## Compatibility

This is connector-internal behavior. The stdout event schema, command schema,
and auth/session file formats are unchanged.
