# puffer-subscriber-telegram-user update 30

The Telegram user subscriber now treats Grammers' persisted MTProto update
state as the authoritative hot-path resume point. `default_init_params` enables
`catch_up` and leaves `update_queue_limit` unbounded so restarts use
`updates.getDifference` before returning to socket updates, while the durable
delivery cursor continues to suppress duplicates and records only messages that
have been delivered or intentionally suppressed.

The previous Puffer-level dialog polling fallback remains removed. Live update
processing reads raw MTProto updates, records lightweight
`update-diagnostics.ndjson` stages, converts raw updates through Grammers, and
persists the Grammers session update state only after the subscriber has handled
the update. This keeps restarts aligned with delivered events instead of
skipping missed messages.

Muted/silent filtering stays cache-only on the live path: socket-delivered
notification-setting updates adjust the cache, and per-message delivery does
not issue remote notification RPCs. Explicit peer listing can still refresh
remote notification settings for user-facing diagnostics.
