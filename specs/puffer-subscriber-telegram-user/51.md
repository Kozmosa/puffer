# Startup Command Preemption

- The Telegram user subscriber now treats startup hydration as interruptible
  setup work after an authenticated session emits `ready`.
- Inbound control commands received during hydration are handled immediately,
  then hydration is retried before the live update loop starts.
- This prevents connector setup actions such as Telegram Desktop import and
  auth checks from timing out behind slow dialog hydration or partial Telegram
  resume failures.
- Replacing the Telegram client during a pre-hydration command still resets the
  account-scoped delivery cursor before hydration resumes, preserving duplicate
  suppression semantics.
