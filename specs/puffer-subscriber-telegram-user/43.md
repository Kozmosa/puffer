# Telegram live update cancellation fix

## Summary

The Telegram user subscriber no longer wraps `Client::next_raw_update()` in an external timeout. The wrapper could cancel Grammers while it was internally performing `updates.getDifference` or `updates.getChannelDifference`, which left the live monitor stalled after the last pushed socket update.

## Behavior

- The live update task now awaits Grammers' update future directly.
- Grammers remains responsible for its own no-update deadlines and difference polling.
- Startup hydration still backfills from `delivery-cursor.json`, so messages missed during a daemon restart are replayed as `resume` events.
- Raw update diagnostics still record `awaiting_raw`, `received_raw`, conversion, and handling stages once an update is returned.

## Compatibility

This preserves the existing stdout event contract, delivery cursor format, notification suppression behavior, and workflow monitor routing. The change only removes a cancellation boundary around the MTProto update hot path.
