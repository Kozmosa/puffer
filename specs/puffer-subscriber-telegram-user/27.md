# Telegram live delivery robustness

The Telegram user subscriber now disables Grammers' bounded update queue by
setting `update_queue_limit` to `None` in the default client init params. The
previous default limit of 100 could drop live updates during bursty chats while
the subscriber was doing per-message mute checks, cursor writes, and stdout
flushes. Dropped updates were later recovered by Telegram difference or Puffer
history catch-up, which made tasks appear minutes late.

The update loop now runs `next_update()` in a dedicated live poller task and
receives updates through an internal channel. The extra Puffer-level dialog
history catch-up scan is no longer run in the live subscriber loop; Grammers'
own update/difference machinery remains responsible for recovering MTProto
gaps. This prevents catch-up scans from blocking live delivery or advancing the
Puffer delivery cursor past messages that should become tasks.

Runtime commands such as peer listing, message search, send actions, or auth
status checks no longer restart the live poller. The poller is restarted only
when the underlying Telegram client is replaced by a reauthentication/import
flow. This prevents updates received during command handling from being queued
on an old receiver and then discarded.

Message events now include `payload.delivery_source` with either `live` or
`catch_up`. Current live subscriber emissions use `live`; the field remains
stable so task history and diagnostics can distinguish true live delivery from
any future replayed messages.
