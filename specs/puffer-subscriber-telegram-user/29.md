# Telegram Live Update State Persistence

The Telegram user subscriber starts live clients from Telegram's current update
state (`catch_up = false`) instead of replaying saved MTProto differences or
running Puffer-owned dialog polling. This keeps the hot path focused on socket
updates and prevents stale session difference state from blocking live delivery.

The session update state is saved only after the subscriber has handled the
socket update, so Puffer does not acknowledge Telegram progress before its own
delivery cursor has advanced during normal live operation.

The live update task synchronizes Grammers' update state into the session after
updates and writes the session file at a short throttle. The update loop now has
only two active inputs: socket-delivered Telegram updates and runtime commands.
It does not scan dialogs or poll message history in the hot path.

Live message emission also avoids remote notification-settings RPCs. Muted
state in the hot path comes from socket notification updates cached by the
subscriber plus each message's silent flag. This keeps one incoming MTProto
message from waiting on `account.getNotifySettings` before it is emitted.

Telegram message payloads include subscriber timing diagnostics:
`subscriber_received_at_ms`, `subscriber_receive_lag_ms`,
`subscriber_emit_at_ms`, `subscriber_emit_lag_ms`, and
`subscriber_queue_lag_ms`. These fields distinguish Telegram/backlog lag from
subscriber queueing and downstream triage delays.
