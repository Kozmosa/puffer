# Continuous live update reader

## Summary

The Telegram personal-account subscriber no longer reconnects the MTProto
client merely because no raw update was received for a fixed idle interval.

## Design

- The live reader now awaits `next_raw_update()` continuously and records raw
  update diagnostics only when the await starts and when Telegram returns an
  update or error.
- Idle periods are treated as normal Telegram behavior, not as stream failure.
- Reconnects remain tied to explicit stream errors, login/client replacement, or
  process restart. Startup hydration still runs at those boundaries.

## Compatibility

This avoids advancing muted-chat delivery cursors during periodic idle
reconnects. Muted chats remain suppressed from Tasks, while truly live messages
can still be observed through message diagnostics.
