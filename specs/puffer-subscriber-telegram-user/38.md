# Single-owner live client updates

## Summary

The Telegram subscriber no longer runs background peer-hash priming RPCs on the
same `Client` that owns the live update stream.

## Design

- Startup hydration still walks dialogs before live streaming begins, which is
  enough to seed dialog state and notification suppression without racing the
  live reader.
- The live reader uses a non-reconnect wake interval around `next_raw_update()`
  so updates queued by command RPCs on the shared client are eventually
  observed. The wake interval is intentionally longer than the vendored
  Grammers difference deadline so it does not cancel the official difference
  fetch path.
- The wake interval does not reconnect, run startup hydration, or advance the
  delivery cursor by itself.

## Compatibility

This keeps muted chats suppressed while preventing concurrent RPC traffic from
processing socket updates that the live reader then fails to observe until a much
later wakeup.
