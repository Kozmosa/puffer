# puffer-skill-evolution

## Purpose

`puffer-skill-evolution` implements the GEPA-style skill generation loop
backing the `/genskill` slash command. It produces reusable SKILL.md
documents from execution traces of a non-trivial task by spawning
multiple candidate sub-agents, scoring them via an LLM-as-judge,
selecting the Pareto frontier, and refining survivors over `K` mutation
rounds.

## Design

The crate is organized around seven modules with one clear
responsibility each:

- `lib.rs` declares the public API: `RubricScores` (with Pareto
  dominance), `GepaOptions` (defaults `N=3`, `K=2`, 15 KB),
  `ExecutionTrace`, `SkillFrontmatter`, `SkillCandidate`, the
  `AgentRuntime` trait, and the `run_gepa` orchestrator.
- `trace.rs` extracts an `ExecutionTrace` from a `Vec<TranscriptStep>`.
  Decoupled from `puffer-session-store::TranscriptEvent` so the crate
  is testable in isolation.
- `parse.rs` parses a `SKILL.md` blob (frontmatter + body) into a
  `SkillCandidate`. Validates that `name` is non-empty lowercase-hyphens
  â‰¤64 chars and `description` â‰¤1024 chars.
- `pareto.rs` computes the non-dominated frontier over scored
  candidates. Pure function, no async, easily tested.
- `judge.rs` invokes an `AgentRuntime` with a rubric prompt to score one
  candidate on four dimensions. Tolerant of JSON in surrounding
  chatter, retries once, defaults to `(0,0,0,0)` on persistent failure.
- `generate.rs` spawns `N` parallel `AgentRuntime::invoke_agent` calls
  with diversity hints, parses each response as a `SkillCandidate`, and
  discards invalid or oversize results. Errors only when zero remain.
- `mutate.rs` produces one mutant per Pareto survivor. The mutation
  prompt instructs the sub-agent to target the survivor's weakest
  dimension while preserving its strong dimensions.

The crate has no I/O dependencies; file writing is the caller's
responsibility (handler in `puffer-core::command_helpers::genskill`).
This keeps the GEPA loop pure and unit-testable.

## Logic

`run_gepa` executes:

1. **Round 0 â€” generation.** Spawn `N` parallel candidate calls; gather
   valid candidates.
2. **Rounds `1..=K`.** For each round, score any unscored candidates
   via `judge.rs`, compute the Pareto frontier via `pareto.rs`. If
   `r < K`, mutate the surviving frontier via `mutate.rs` and union the
   mutants into the candidate pool.
3. **Final selection.** From the last frontier, pick the candidate with
   the highest `reproducibility`. Tie-break on the sum of all four
   scores.

Every public function carries `///` documentation per repo convention,
and no source file exceeds the 1000-line limit.

## Contract

`AgentRuntime` is the testability seam:

```rust
#[async_trait::async_trait]
pub trait AgentRuntime: Send + Sync {
    async fn invoke_agent(&self, prompt: &str) -> anyhow::Result<String>;
}
```

Production code in `puffer-core` wraps the live agent dispatcher; tests
provide canned-response implementations.

`run_gepa` errors only when Round 0 returns zero candidates (every
generation attempt failed validation). Subsequent rounds tolerate
mutation and judging failures gracefully â€” a missing mutant simply
keeps the Pareto survivor in the pool unchanged.

`SkillCandidate::scores` is `Option<RubricScores>` â€” `None` means "not
yet judged". `pareto_frontier` excludes unscored candidates.

The frontmatter contract enforces:

- `name`: non-empty, â‰¤64 chars
- `description`: â‰¤1024 chars

Body size is bounded by `GepaOptions::max_size_bytes` (default 15 KB);
oversize candidates are discarded by `generate` and `mutate`.

The Pareto contract: a candidate is on the frontier iff no other
scored candidate has `>=` on all four dimensions and `>` on at least
one.

The tie-break contract: when multiple Pareto candidates remain, prefer
the highest `reproducibility`, then the highest total score.

## Architecture Details

The crate follows puffer's flat-file convention (no `src/` subdir).
`Cargo.toml` declares `path = "lib.rs"` explicitly; modules are at
`crates/puffer-skill-evolution/<module>.rs`.

Async dispatch uses `tokio` (workspace dependency, `sync`/`macros`
features). `async-trait` is required because `AgentRuntime` has async
methods on a trait object.

The crate intentionally takes no dependency on `puffer-core`,
`puffer-session-store`, or `puffer-resources`. Calling into it from a
puffer-core handler means converting transcript events into the
crate's local `TranscriptStep` type at the boundary, which keeps the
GEPA logic provider- and storage-agnostic.
