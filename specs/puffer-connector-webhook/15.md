# Opsgenie Webhook Normalizer

The webhook connector now recognizes outgoing Opsgenie alert action callbacks
and converts them into provider-specific inbound messages before the generic
webhook fallback.

## Design

- Payload detection is conservative: Opsgenie handling requires an `alert`
  object, an action, a readable alert message or description, and either
  Opsgenie alert identity fields (`alertId` plus alias/tiny/user metadata), an
  integration hint, or the optional `X-Opsgenie-Webhook` header.
- Official callback and Edge Connector shapes are supported, including nested
  `payload.alert` wrappers.
- Conversation ids prefer stable identifiers in this order:
  `opsgenie:alert:<alert-id>`, `opsgenie:alias:<alias>`,
  `opsgenie:tiny:<tiny-id>`, `opsgenie:integration:<integration-id>:<action>`,
  then `opsgenie:event:<action>`.
- Message summaries include action, integration, message, alert id, alias, tiny
  id, priority transitions, entity, source, user, tags, teams, recipients,
  description, details, and created or updated timestamps when present.

## Compatibility

The shape guard rejects generic `{ action, alert }` payloads that lack
Opsgenie-specific alert identity. Existing provider normalizers keep their
precedence because Opsgenie is evaluated after New Relic and before PagerDuty.
