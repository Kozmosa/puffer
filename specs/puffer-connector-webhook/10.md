# PagerDuty Webhook Normalization

## Change

The generic webhook connector now recognizes PagerDuty V3 incident, service, and
test deliveries and converts them into `InboundMessage` values before falling
back to later provider-specific normalizers.

## Recognition

A payload is considered PagerDuty-shaped when it carries PagerDuty V3 request
headers such as `PagerDuty-Webhook/V3.0`, `x-webhook-subscription`, or
`x-pagerduty-signature`. Headerless payloads must expose a top-level `event`
object with PagerDuty-specific event types or resource hints such as
`incident`, `service`, or the Pagey test event.

## Conversation Routing

Conversation ids are stable by service and subject:

- `pagerduty:<service>:incident:<incident-id>`
- `pagerduty:<service>:service:<service-id>`
- `pagerduty:<service>:event:<delivery-id>`

If no subject id exists, the connector falls back to the delivery id or event
type.

## Message Summary

Summaries include the event type, resource type, service, subject, status,
urgency, priority, URL, occurrence time, delivery id, and short message body
when those fields are present.

## Security

The normalizer records PagerDuty signature headers only as a shape hint.
Signature verification is intentionally not implemented yet because the
deterministic setup flow does not collect the PagerDuty webhook signing secret.
