# Sentry Webhook Normalization

## Change

The generic webhook connector now recognizes Sentry Integration Platform and
service-hook style JSON deliveries and converts them into `InboundMessage`
values before falling back to later provider-specific normalizers.

## Recognition

A payload is considered Sentry-shaped when it carries a Sentry webhook header
or an integration installation object with delivery data. Headerless payloads
must expose a Sentry-specific `data.issue`, a Sentry-specific `data.event`, or
alert rule fields such as `triggered_rule`; generic `data.event` JSON is left
for later normalizers instead of being claimed as Sentry.

## Conversation Routing

Conversation ids are stable by project and subject:

- `sentry:<project>:issue:<issue-id>`
- `sentry:<project>:event:<event-id>`
- `sentry:<project>:alert:<alert-id>`

If no subject id exists, the connector falls back to resource plus delivery id
or action.

## Message Summary

Summaries include the resource, action, project, alert rule, subject, status,
level, culprit, environment, URL, event id, delivery id, and short message body
when those fields are present.

## Security

The normalizer records Sentry signature headers only as a shape hint. Signature
verification is intentionally not implemented yet because the deterministic
setup flow does not collect the Sentry integration secret.
