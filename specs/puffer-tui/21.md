# Async shell shortcut execution

## Goal

Interactive `!cmd` shell shortcuts can take seconds or minutes. Running them
synchronously from queued prompt dispatch blocks the TUI event loop and makes
the app look frozen.

## Design

Composer-submitted shell shortcuts now start a pending submit backed by a worker
thread. The UI records the user-visible `!cmd` message immediately, keeps the
shell output semantics unchanged, and polls a completion event before appending
the assistant or system reply. Non-interactive direct submit paths keep the
existing inline behavior.

## Contracts

- Shell shortcuts submitted during a running turn are still queued in order.
- Starting a queued shell shortcut creates a pending submit instead of blocking
  the TUI loop.
- Shell stdout/stderr transcript rendering keeps the previous success/failure
  behavior.
- Tool hooks still receive `tool_start` and `tool_end` events around the shell
  execution.
