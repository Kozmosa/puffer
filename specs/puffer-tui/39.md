# Transient Resume Picker Session Persistence

## Problem

The CLI startup resume picker now opens without creating a persisted session.
That avoids blank history rows, but the TUI still needs a real session before it
can write transcript events if the user dismisses the picker and starts a new
conversation.

## Design

- `Uuid::nil()` marks the startup resume picker as an in-memory session only.
- Selecting a scoped `/resume <uuid>` from the picker keeps the transient state
  until the target session is loaded, so no blank session is inserted.
- New prompt, shell, loop, and ordinary command submissions lazily call
  `create_session` before the first transcript write.
- `/resume` with no arguments remains an overlay-only prompt path, so reopening
  the picker from the composer does not require persistence.

## Test Coverage

- `transient_resume_picker_selection_does_not_create_blank_session` verifies a
  picker selection resumes the existing session without adding history rows.
- `transient_resume_picker_new_prompt_creates_session_lazily` verifies a new
  prompt from transient state gets a real persisted session before writing.
