# Permission Prompt Interrupts

## Context

The global `Ctrl+C` interrupt path cancels an in-flight provider turn, but
permission prompt overlays handled the same key before the global path could see
it. During an active turn this denied the permission request without cancelling
the worker, so the provider turn could continue running.

## Change

- When a permission prompt is visible and a provider turn is pending, `Ctrl+C`
  now cancels the pending turn before prompt-specific key handling.
- The stale permission prompt and pending response channel are cleared when the
  turn is interrupted.
- Loop turns preserve the existing `Ctrl+C` loop-stop behavior.

## Compatibility

Permission prompts still treat `Ctrl+C` as deny when no provider turn is
pending. The change only affects prompts attached to an active pending turn.

## Verification

- Added a TUI regression that drives the full `handle_key` path with a pending
  turn and permission overlay, then verifies the cancel token is tripped and no
  permission response is sent.
