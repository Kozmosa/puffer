# TUI Unscoped Model Validation

## Problem

The TUI onboarding gate accepted any `current_model` string without a provider
prefix. That let stale values such as `gpt-unknown` skip provider/model setup
even when the active provider did not expose that model.

## Design

- Keep accepting unscoped model ids for compatibility with the active provider.
- Validate unscoped ids against the current provider's model list.
- Re-enter provider/model setup when the unscoped id is not available.
- Preserve the existing provider-prefixed `provider/model` validation path.

## Test Coverage

- `initial_provider_setup_accepts_unscoped_current_provider_model` covers the
  compatibility case where the current provider owns the unscoped id.
- `initial_provider_setup_rejects_unknown_unscoped_model` covers stale or
  misspelled unscoped ids that should reopen setup.
