# Contact-Scoped Connector Streams

- Contact ids are normalized as `<prefix>@<connector identity>` with standard prefixes
  for Telegram, Google/email/calendar, Slack, Discord, Matrix, and Lark.
- Connector event payloads can expose explicit `contact_id` or `contact_ids`; helper
  extraction also derives ids from known connector fields while requiring service
  hints for generic keys such as `sender_id`, `author_id`, and `user_id`.
- The connector protocol supports contact listing, contact search, contact context,
  and contact-scoped subscribe commands. Empty subscribe `contact_ids` preserves the
  legacy all-contact stream behavior.
- Workflow bindings carry normalized `contact_ids`. The manager passes the union of
  enabled binding ids for a connection into the source stream, unless any enabled
  binding is all-contact; in that case the source stream is intentionally unfiltered.
  A running command-backed stream is restarted when that effective source contact
  filter changes.
- Command-backed connectors can expose one-shot `contacts` operations. The manager
  sends list/search/context JSON commands on stdin and parses contact/contact-context
  frames on stdout.
- The router still drops non-matching events per binding before regex/classifier/action
  work, so source filtering is only an optimization and not the authority for scope.
- Invalid contact ids are rejected during binding validation. Legacy persisted
  bindings deserialize with an empty contact list and remain compatible.
