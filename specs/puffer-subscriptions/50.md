# Blocking router action dispatch

## Scope

Subscription router event dispatch now runs workflow action processing on a
Tokio blocking thread instead of directly inside the async router task.

## Design

- `SubscriptionRouter::run` still receives events from the async bus and keeps
  control-event filtering plus `events_seen` accounting in the router loop.
- Non-control envelopes are moved into `process_envelope_blocking`, which clones
  the store, history store, dispatcher, classifier, and stats handles and calls
  `process_envelope_result` inside `tokio::task::spawn_blocking`.
- The blocking path preserves existing matching, classification, action
  dispatch, history persistence, and acted/failed stat updates.
- If the blocking task itself panics or is cancelled, the router records one
  failed event and logs the join error instead of taking down the router loop.

## Compatibility

- Event ordering remains serial because the router awaits each blocking dispatch
  before receiving the next event, matching the prior synchronous behavior.
- Workflow triage actions may create or drop their own runtimes without
  panicking the subscription runtime's async worker.
- A regression test covers dispatchers that create and drop a Tokio runtime
  during action execution, which matches the live monitor triage failure mode.
