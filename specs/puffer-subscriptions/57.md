# Router event intake robustness

The subscription router now drains subscriber events independently from action
execution. Each non-control envelope increments router intake counters
immediately, then runs workflow matching and action dispatch in a bounded
background processor pool.

This prevents a slow triage agent, connector action, or workflow run from
blocking the router's broadcast-bus receiver. The bus remains bounded, but the
router no longer lets ordinary action latency consume that buffer and drop
newer monitor events during bursts.

Action execution remains bounded by a fixed processor semaphore. Events beyond
the concurrency limit wait in lightweight Tokio tasks rather than in the
broadcast receiver, preserving delivery order best-effort without blocking
subscriber stdout intake.

The router also checks workflow history for an existing `(workflow_slug,
dedup_key)` before dispatch. Subscriber replays and at-least-once delivery
retries therefore do not create duplicate monitor tasks once an earlier run or
router skip has been persisted.
