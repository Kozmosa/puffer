# puffer-subscriptions update 67

Nested message sender display names now use the same email-header cleanup as
top-level `from` fields. Payloads such as `message.sender:
Service <robot@example.net>` produce `Service`, while a bare
`message.sender: robot@example.net` leaves the contact name unset.

Telegram-style display-name extraction also treats `chat_title` as a contact
name only for direct user chats, where the chat identity is the contact. Group
chat titles are not used as sender contact names; group contacts still use
`sender_name` when it is available.

This matches the existing contact-id extraction contract, where
`message.sender` is already treated as an email-bearing payload field, and keeps
subscription-backed Telegram history aligned with daemon contact history.
Contact ids, filtering semantics, and connector ownership rules are unchanged.
