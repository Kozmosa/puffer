# Queued Monitor Triage Batching

## Summary

Connector streams with an installed processor now serialize processing through
one per-connection queue. The stream reads connector stdout continuously, keeps
events pending while the current processor turn is running, and acks connector
cursors only after the corresponding processed batch succeeds.

## Design

- `ConnectorEventProcessor` keeps the one-event method and adds a batch method
  with the old sequential behavior as the default implementation.
- `ConnectorStreamHandle` creates an unbounded pending-event channel when a
  processor is installed. The processor task takes the first pending event,
  drains any events that arrived while it was busy, and processes that batch in
  one blocking call.
- Successful batches emit one ack per event and advance the stored cursor in
  connector order. Failed batches stop the stream without acking, so restart
  resumes from the last acknowledged cursor.
- The subscription router exposes `process_envelope_batch_result` for
  same-connection batches. It applies the same paused, topic, dedup, monitor
  muted, ignore-filter, trigger-filter, and classifier gates as the single-event
  path.
- Triage-agent actions are grouped per matching binding and dispatched once for
  the batch. Non-triage actions keep one dispatch per event.

## Compatibility

Existing single-event processors and dispatchers still compile through default
trait methods. Connectors without a processor continue publishing and acking
events immediately. History rows are still written per event, even when the
underlying triage turn handled multiple events.
