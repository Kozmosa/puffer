# Provider Alias Canonicalization

## Summary

The provider registry now recognizes desktop-facing provider aliases during
provider and model lookup.

## Design

`canonical_provider_id` maps `codex` and `openai` to `openai`, and `claude`
and `anthropic` to `anthropic`. `ProviderRegistry::provider` and
`provider_entry` still try exact ids first, then fall back to the canonical
built-in alias.

## Contracts

- Exact provider ids remain authoritative for registered providers.
- Built-in aliases resolve when no exact provider id exists.
- `resolve_model("codex/gpt-5")` can resolve models from the canonical
  `openai` provider.
- Unknown provider ids are trimmed and otherwise left unchanged.

## Verification

- `cargo test -p puffer-provider-registry registry_resolves_desktop_provider_aliases`
