# Discovery Alias Canonicalization

## Summary

Single-provider model discovery now accepts the same desktop-facing aliases as
normal provider and model lookup.

## Design

`ProviderRegistry::discover_and_merge_provider` canonicalizes the requested
provider id before reading registry entries, checking auth, or merging
discovered models. This keeps `/model refresh codex`, `/model refresh claude`,
and onboarding best-effort refreshes aligned with the existing lookup behavior
documented in `02.md`.

## Contracts

- Exact provider ids and unknown custom provider ids are still trimmed and
  otherwise preserved.
- `codex` refreshes the canonical `openai` provider and uses `openai`
  credentials.
- `claude` refreshes the canonical `anthropic` provider and uses `anthropic`
  credentials.
- Providers without discovery configured remain a successful no-op after alias
  resolution.

## Verification

- `cargo test -p puffer-provider-registry registry_resolves_desktop_provider_aliases`
- `cargo test -p puffer-provider-registry parse_openai_discovery_response_maps_models_through_codex_alias`
