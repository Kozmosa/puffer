# puffer-media 01: Declarative media generation engine (image + video)

- Date: 2026-06-11 (revised 2026-06-11: gap-fill + over-design trim; rewritten in English)
- Status: design agreed, implementation plan pending
- Constraints: **no backward compatibility**; optimize only for long-term value, stability, performance; **avoid over-design**

## 1. Background and problem

Today each media provider is a hand-written full-stack adapter (build body + auth + poll +
parse + download). Comparing the three real request shapes (byteplus / relaydance / kling),
the differences concentrate in 4 points — **auth, request encoding, lifecycle, response
parsing** — and the other ~80% is duplicated. Concrete problems found in the code:

- The media layer uses `reqwest::blocking` while the host runtime is tokio async — an impedance mismatch.
- Each in-flight render holds one blocking thread, `thread::sleep`-polling for the full render duration.
- `Client::new()` is scattered across adapters; no connection-pool reuse.
- No idempotency: retries / network jitter can double-submit and double-charge.
- Crash recovery relies on a best-effort "reclaim on the next generate call", not proactive.
- Dispatch is split across two `match adapter.as_str()` sites (`video.rs` / `runtime.rs`) plus a
  `MediaExecutionKind` enum; adding one provider touches 6 files.
- Image path is synchronous, video path is asynchronous — two brains.

Mainstream references: fal (uniform queue, sync = degenerate async, per-model output schema,
idempotent request_id, auto-retry); LiteLLM (provider config + transform_request/
transform_response hooks + unified error mapping); Vercel AI SDK (uniform provider spec +
capability metadata).

## 2. Goals

1. **Extensible**: adding a provider/model on a known protocol = edit YAML, zero Rust.
2. **Stable**: crash-recoverable, retries never double-charge, errors handled by class.
3. **Performant**: connection-pool reuse, no scattered clients, polling with jittered backoff.

## 3. Key decisions (agreed)

| Area | Decision | Rationale |
|---|---|---|
| Concurrency | **Keep blocking + one global shared Client + one background reconciler thread (orphan recovery only)** | ~90% of the perf/stability gain without the regression surface of a full async migration. Avoids over-design. |
| Call contract | **Happy path still awaits terminal inline and returns the finished artifact** (agent UX unchanged); the reconciler does not steal polling | Returning immediately would leave the agent without a result; minimal change, UX intact. |
| Declarative depth | **Medium**: the `execution` block stays at the «provider + kind» level (matching today's structure), enriched with field mapping / status vocabulary / result pointers. **No** cross-provider preset-inheritance layer | At 3-4 providers, an inheritance/merge layer is over-design and a bug surface; the schema leaves the door open to extract presets when real duplication appears. |
| Stability scope | **In**: persist task_id on submit; reconciler recovers orphans; 3-way error class drives retry; idempotency id (`external_task_id` where supported, local dedup otherwise). **Out**: Webhook | Webhook needs a public endpoint the desktop app lacks — YAGNI. The rest are real stability wins. |
| Auth | Add `StoredCredential::AccessKeySecret{access_key, secret_key}` + `AuthMode::AccessKeySecret` | No compat burden; JWT becomes first-class, cleaner than packing `ak\|sk` into ApiKey. |
| Sync image | Modeled as `lifecycle: inline` (terminal on submit), folded into the same engine | Removes the image/video two-brain split. |
| Quirk handling | **Zero quirks in v1**: byteplus numeric `duration` via axis `wire_type: number`, `generate_audio:false` via `constants`. `quirks/` is a documented escape hatch, not built | All three providers need no quirk; build it only if one appears. |
| Type placement | Execution-spec types live on `MediaExecutionDescriptor` in **puffer-provider-registry** (catalog data); behavior (codec/engine/reconciler) lives in **puffer-media** | Data types belong with the catalog; avoids duplicating types across crates. |
| Delivery order | **Strangler**: migrate existing providers first (golden parity) → Kling last as the extensibility proof | Prove the engine against known-good behavior before adding new functionality; delete superseded code each step; no long dual-system window. |

Two rejected extremes: full async + cross-provider preset DSL (over-design); add Kling without
restructuring (accrues debt).

## 4. Architecture overview

```
Tool layer   Video/ImageGeneration tool                               (unchanged)
  │   one call returns the finished artifact (inline await)
Selection    resolver + discovery (kept): logical model + axes → ResolvedMediaRequest
  │     { kind, provider_id, model_id, params(by request_field), numeric_fields,
  │       prompt, image_refs, client_request_id }
  │
Asset stage  asset:// → url or base64 (per spec.image_refs.encoding)   (new, pre-submit)
  │
Engine       MediaEngine (consumes &MediaExecutionDescriptor looked up by provider+kind)
  │     ├ submit(...)        → MediaJob (task_id persisted)
  │     ├ await_terminal(...) → inline poll to terminal (happy path)
  │     ├ AuthStrategy   Bearer | JwtHs256(ak/sk)
  │     ├ RequestCodec   build body from execution.request
  │     └ ResponseCodec  parse via execution.response (JSON pointers + status_map)
  │
Reconcile    Reconciler (one background worker: at daemon boot + slow interval)
  │     adopts only "orphan" jobs (non-terminal, no live owner) → poll to terminal → output
  │     re-resolves the execution spec from job.provider_id + job.kind
  │     jittered backoff; 3-way error class; one job's failure never blocks others
  │
Base         MediaJob state machine / sidecar (kept) + global shared reqwest::blocking::Client (pool)
```

**Two clearly separated paths:**
- **Happy path**: tool → engine.submit (persist task_id) → engine.await_terminal (inline poll) → output → return artifact. Same behavior as today; only the engine underneath changes.
- **Recovery path**: the reconciler only runs at daemon boot + a slow interval, adopting **orphans** (non-terminal jobs whose `owner` is no longer live), resuming already-paid tasks. This avoids double-polling against the inline waiter.

## 5. Module boundaries

Each unit must: be unit-testable (injectable transport); be understandable without reading its
internals; allow internal changes without breaking callers.

| Module | Responsibility | Depends on |
|---|---|---|
| `puffer-provider-registry/src/model.rs` (modify) | Extend `MediaExecutionDescriptor` with `protocol/auth/request/response` + `validate()` | serde |
| `puffer-provider-registry/src/auth.rs` (modify) | `AccessKeySecret` credential + `AuthMode` variant | serde |
| `puffer-media/.../auth.rs` (new) | `AuthStrategy::{Bearer, JwtHs256}`; mint + apply | hmac/sha2/base64 |
| `puffer-media/.../request_codec.rs` (new) | Build submit body from `request` spec + resolved request | registry types |
| `puffer-media/.../response_codec.rs` (new) | Parse submit/poll via `response` spec (RFC6901 pointers, status_map, url/b64, error class) | registry types |
| `puffer-media/.../transport.rs` (new) | `MediaTransport` trait + reqwest impl + fake | http |
| `puffer-media/.../engine.rs` (new) | `submit()` + `await_terminal()` (SubmitPoll / Inline) | the above + jobs |
| `puffer-media/.../reconciler.rs` (new) | Orphan selection + `run_once` | engine/jobs |
| `puffer-media/.../output.rs` (new) | url download / b64 decode → artifacts; video → poster; N artifacts for batch | http + jobs + video_poster |
| `puffer-media/.../http.rs` (new) | Global shared Client, timeouts, URL safety, secret redaction | reqwest |

> `quirks/` is **not built in v1** (all three providers need none). It stays a documented escape
> hatch: if a future provider has a quirk that cannot be expressed declaratively, add
> `trait ProviderQuirks { fn after_build_body(&self, body, r) }`.

Interface sketch (intent, not final):

```rust
trait AuthStrategy { fn apply(&self, req: RequestBuilder, now_s: u64) -> Result<RequestBuilder>; }
trait RequestCodec { fn submit_body(&self, r: &ResolvedMediaRequest, prompt: &str, image_refs: &[String]) -> Result<Value>; }
trait ResponseCodec {
    fn parse_submit(&self, v: &Value) -> Result<SubmitOutcome>; // task_id | inline result
    fn parse_poll(&self,   v: &Value) -> Result<PollState>;     // status + results(url|b64) + error class
}
enum Lifecycle { SubmitPoll, Inline }
enum ErrorClass { Retryable, Terminal, Auth }
```

## 6. Data model

### execution block (provider+kind level, shared by all that provider's models of that kind)

Kling video example:

```yaml
media:
  video:
    discovery: { adapter: static }          # discovery layer unchanged (see §6.3)
    execution:
      protocol: submit_poll_task            # submit_poll_task | inline
      auth: { mode: jwt_hs256 }
      base_url: https://api-singapore.klingai.com
      endpoint: /v1/videos/text2video
      endpoint_with_image: /v1/videos/image2video
      request:
        model_field: model_name             # default "model"
        prompt: { mode: field, field: prompt }        # field | content_array
        image_refs:                                   # unsupported | content_image_url | named_fields
          mode: named_fields
          fields: [image, image_tail]
          encoding: url                                # url | base64
        idempotency_field: external_task_id
        constants: {}                                  # fixed fields, injected before params
        # param placement reuses each axis's request_field; dotted path nests, e.g. "metadata.ratio"
      response:
        task_id_pointer: /data/task_id
        status_pointer: /data/task_status
        status_map: { submitted: queued, processing: running, succeed: succeeded, failed: failed }
        result:
          kind: url                                    # url | b64
          pointer: /data/task_result/videos            # array or single value
          url_field: url                               # when kind=url, take this field from each element
        error_pointer: /data/task_status_msg
        retryable_http: [429, 500, 502, 503, 504]
    models:                                  # models carry only axes + variants (as today)
      - id: kling-v2-1
        operations: [generate]
        axes: [ ... mode/duration/aspect_ratio ... ]
        variants: { model_name: kling-v2-1 }
```

All three providers + image fit this table (proof it is sufficient):

| Field | byteplus video | relaydance video | kling video | kling image | openai image |
|---|---|---|---|---|---|
| protocol | submit_poll_task | submit_poll_task | submit_poll_task | submit_poll_task | inline |
| auth | bearer | bearer | jwt_hs256 | jwt_hs256 | bearer |
| model_field | model | model | model_name | model_name | model |
| prompt.mode | content_array | field | field | field | field |
| image_refs.mode | content_image_url | unsupported | named_fields | named_fields(image) | — |
| constants | generate_audio:false | n:1 | — | — | — |
| param nesting | flat | metadata.* | flat | flat | flat |
| result.kind | url | url | url | url | url or b64 |
| result.pointer | task→video_url | same | task_result/videos | task_result/images | data |

### 6.1 Encoding interactions (request_codec must make explicit)

- **content_array + content_image_url** (byteplus): `content = [ {type:text,text} ] + [ {type:image_url,image_url:{url}} ... ]`; prompt and images merge into one content array.
- **constants ordering**: `constants` are injected before param fields, so a future param axis can override a same-named constant (matches the byteplus `generate_audio` note).
- **numeric encoding**: a param is emitted as a JSON number when its axis `wire_type == number`. The resolver surfaces these as `numeric_fields` on `ResolvedMediaRequest` (axes are dropped after resolution, so the set must travel with the request).

### 6.2 Batch (image n>1)

Reuse the existing `batch` descriptor (`per_image` / `exact`):
- `per_image`: engine submits n times, aggregates n artifacts (current semantics).
- `exact`: single request returns multiple images; output writes n artifacts.
- Video keeps `count == 1` (existing validation retained).

### 6.3 Discovery layer (kept, not rewritten)

`discovery` (`static` / `trusted_image_output` dynamic discovery) and `resolve_media_capabilities`
are kept as-is. This change replaces only the execution/adapter layer; capability enumeration,
dynamic image-model discovery, and auth-based filtering are untouched. (Dynamically discovered
image models needing an execution spec is an Increment 2 concern.)

### 6.4 MediaJob additions (on top of existing struct)

`client_request_id`, `provider_task_id` (exists), `next_poll_at_ms`, `attempt`, `error_class`,
`owner` (live-waiter id, for orphan detection). The old `adapter` routing tag is retired; the
reconciler re-resolves the execution spec from `provider_id` + `kind`.

## 7. Data flow

- **submit_poll_task**: resolver+discovery → asset stage → engine.submit (auth/encode/POST) → ResponseCodec extracts task_id → persist (queued, owner = this call) → **engine.await_terminal inline poll** → terminal → output (download/decode; video adds poster) → return artifact.
- **inline** (sync image): engine.submit gets the result directly → output → return (job created already-succeeded).
- **recovery**: reconciler adopts only non-terminal orphan jobs with no live owner.

## 8. Error handling

- 3-way class: `Retryable` (network / `retryable_http` / 429) → backoff + jitter retry (bounded by `attempt`); `Terminal` (4xx / validation / moderation) → fail immediately, stop polling; `Auth` (401 / expired JWT) → mark needs-reauth. `await_terminal` honors this class — it does not blindly retry.
- Unknown **task_status** (not an HTTP-level error) keeps the existing conservative philosophy: treat as running, keep polling.
- All error strings pass through `redact_secrets`; one job's failure in the reconciler never blocks others.
- Backoff is jittered (avoid synchronized retries).

## 9. Testing strategy

- **Contract fixtures**: per provider, golden request body + poll responses (json), replayed through the engine (reuse the existing `Fake*Transport` injection pattern).
- When migrating byteplus/relaydance, reuse their existing fixtures to lock behavior parity (regression baseline).
- Cover: 3-way error class, inline vs submit_poll, batch per_image, url vs b64 results, content_array+image_url merge, orphan recovery, numeric encoding.
- **Adding a provider = add a fixture + execution block; engine code unchanged.**

## 10. Delete / keep / not-doing

- **Delete**: the full-stack logic in `byteplus_video.rs` / `relaydance_video.rs` / `replicate_video.rs` / `images_json.rs` / `minimax_image.rs` / `chat_image_output.rs`; the `MediaExecutionKind` enum; the dual `match adapter` in `video.rs`+`runtime.rs`; `reject_unsupported_video_image_references`; scattered `Client::new()`; "reclaim on next generate". `replicate_video` has no real provider (only the kling placeholder + daemon test refs) → pure deletion + test fix-up, no migration.
- **Keep**: resolver axis/variant layer; **discovery layer**; MediaJob/artifact/sidecar; video_poster; URL safety / redaction; injection-based test pattern; batch semantics; video count==1 validation.
- **Not doing**: cross-provider preset inheritance; Webhook; full async migration; no `quirks/` in v1.

## 11. Scope and phasing (each step compiles and tests independently)

1. Base: `http.rs` shared client + MediaJob field extensions (owner/attempt/...).
2. Engine: extend `MediaExecutionDescriptor` with the spec types + `validate()`; auth / request_codec / response_codec + engine.submit + engine.await_terminal (with error class).
3. Migrate byteplus + relaydance video to the engine (golden parity), delete old.
4. Migrate image (openai / minimax / chat) to inline, incl. batch and url/b64, delete old.
5. Add Kling: video (t2v/i2v) + image = execution blocks + JwtHs256 + AccessKeySecret credential.
6. Build the reconciler (orphan-recovery core + daemon scheduling); retire `MediaExecutionKind`, the dual dispatch, and the replicate placeholder.

The reconciler lands in Phase 6 (not earlier): the inline happy path needs no reconciler, and an
orphan-recovery core with no scheduling is unused infrastructure. Delivering core + scheduling
together — once the "which workspace job stores" question is resolved — avoids building it twice.

## 12. Risks and boundaries

- Reconciler ownership = daemon; must specify: which workspace job stores it scans, boot + slow-interval trigger, graceful stop, and `owner`/heartbeat-based orphan detection to avoid double-polling. The whole reconciler (orphan-recovery core + daemon scheduling) is delivered together in **Phase 6**, once the workspace-source question is resolved — not built earlier as unused infrastructure.
- CLI one-shot (no daemon): the inline happy path still works; orphan recovery only applies when a daemon is running (acceptable).
- Idempotency depends on each provider supporting `external_task_id`: Kling supports it; byteplus/relaydance need their equivalent confirmed, else degrade to "local dedup + pre-submit lookup".
- JSON pointers support RFC6901 + "take url_field from array elements"; **no** full JSONPath. Execution-spec deserialization must run `validate()`, backed by contract tests, to prevent silent misconfiguration.
- Kling i2v only accepts public `https://` or raw base64; `asset://` is resolved to url/base64 in the asset stage, else an explicit error.
- No pure DSL: a real quirk is the only trigger for the `quirks/` escape hatch.
