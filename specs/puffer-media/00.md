# puffer-media 00: Media Runtime Crate Split

## Goal

Move the pure image/video generation runtime out of `puffer-core` into a new
`puffer-media` crate so media changes can be tested through Cargo's package
boundary instead of compiling and enumerating all `puffer-core` tests.

The target day-to-day command for media runtime changes is:

```sh
cargo test -p puffer-media
```

No wrapper script is part of this design. In particular, do not add
`scripts/test-media.sh`.

## Current Baseline

The media runtime currently lives inside `puffer-core`:

- `crates/puffer-core/runtime/media/**`: 7,825 lines across adapters,
  persistence, resolver, poster extraction, and module tests.
- `crates/puffer-core/media_runtime.rs`: 505 lines of public facade and DTOs.
- `crates/puffer-core/media_runtime_artifacts.rs`: 688 lines of generated media
  preview, timeline attachment, and video access helpers.
- `crates/puffer-core/media_runtime_internal_tools.rs`: 76 lines of Bash helper
  output parsing.
- `crates/puffer-core/media_runtime_video.rs`: 461 lines of exact video routing
  and orphan reclamation.

`cargo test -p puffer-core --lib media -- --list` currently lists 148
media-named tests, but it still compiles the full `puffer-core` lib and catches
workflow/system-prompt tests that only contain `media` in the path or name.
Unscoped `puffer-core` test filtering is also fragile because explicit
integration targets in `crates/puffer-core/Cargo.toml` remain part of package
test selection.

## Decision

Create `crates/puffer-media` and move only the media runtime/domain layer into
it.

Move into `puffer-media`:

- exact media facade, request/result DTOs, discovery cache, and capability view;
- generated media preview, video access metadata, and timeline attachment
  helpers;
- internal media Bash output parsing helpers;
- image/video adapters under the current `runtime/media` module;
- media job/artifact persistence, path validation, resolver, planner, poster
  extraction, fixtures, and their tests.

Keep outside `puffer-media`:

- `puffer-core/runtime/claude_tools/workflow/image_generation.rs`;
- `puffer-core/runtime/claude_tools/workflow/video_generation.rs`;
- `puffer-core/runtime/internal_tool_permissions.rs`;
- `puffer-core` tool dispatch, `AppState`, runtime prompt, and permission state;
- `puffer-cli` daemon RPC DTOs, desktop timeline DTO conversion, and generated
  video ticket serving;
- `puffer-resources` provider YAML, skill text, internal tool resources, and
  schema/governance tests;
- `puffer-tools` internal tool registry behavior.

Do not keep compatibility re-exports from `puffer-core`. Consumers must import
media APIs from `puffer_media`.

## Dependency Contract

`puffer-media` is a lower-level runtime crate. Production dependencies may
include shared third-party crates and `puffer-provider-registry`. Production
dependencies must not include:

- `puffer-core`;
- `puffer-cli`;
- `puffer-tools`;
- `puffer-resources`;
- `puffer-tui`;
- `puffer-workflow`.

`puffer-resources` may be used only as a dev-dependency if tests truly need
resource pack parsing. Prefer direct `ProviderDescriptor` construction or
crate-local YAML fixtures for media runtime tests so production dependency
direction stays clean.

Expected production direction:

```text
puffer-provider-registry -> puffer-media -> puffer-core -> puffer-cli
```

This is a logical layering rule: `puffer-media` consumes provider descriptors
and auth stores, while `puffer-core` consumes media runtime APIs.

## Alternatives Considered

### Keep Everything In `puffer-core`

This is the lowest migration cost, but it does not solve the compile/test blast
radius. Filtered commands still compile `puffer-core`, bring in unrelated
runtime warnings, and can accidentally include package integration targets.
Reject.

### Split `puffer-media` Runtime Only

This gives a stable Cargo package boundary around the code that changes when
image/video generation behavior changes. It preserves current user-visible
flows and keeps workflow/daemon concerns in their existing crates. Accept.

### Split Image And Video Into Separate Crates

Image and video share provider resolution, auth helpers, job/artifact sidecars,
preview metadata, request planning, and path safety checks. Splitting them now
would duplicate these contracts or introduce an extra shared crate. Reject for
now as overdesigned.

### Move Workflow Tools And Daemon RPCs Into `puffer-media`

The workflow tools depend on `AppState`, config, permission/runtime dispatch,
and tool output conventions. The daemon depends on HTTP DTOs, session lookup,
and generated video tickets. Moving those would turn `puffer-media` into an app
runtime crate and recreate the current coupling under a new name. Reject.

## Migration Plan

1. Add crate skeleton.
   - Add `crates/puffer-media` to workspace members.
   - Use a normal `src/lib.rs` layout for the new crate.
   - Add production dependencies needed by the moved runtime:
     `anyhow`, `base64`, `reqwest`, `serde`, `serde_json`, `uuid`, and
     `puffer-provider-registry`; add `tempfile`, `indexmap`, and `serde_yaml`
     only as dev-dependencies when tests require them.

2. Move media runtime files.
   - Move `crates/puffer-core/media_runtime*.rs` into `puffer-media` facade
     modules.
   - Move `crates/puffer-core/runtime/media/**` into `puffer-media/src/media/**`.
   - Move `runtime/media/fixtures/*.json` with the adapter tests so
     `include_str!("fixtures/...")` remains valid.
   - Rename imports from `crate::runtime::media::*` to `crate::media::*` or
     module-local imports.

3. Stabilize the `puffer-media` facade.
   - Keep internal modules private.
   - Re-export only the public API listed above.
   - Keep `ExactMediaDiscoveryCache::from_inner_for_test` crate-private for
     `puffer-media` tests; do not make `puffer-core` tests depend on
     `puffer-media` internals.
   - Remove or rewrite cross-crate tests that construct resolver internals.

4. Update `puffer-core`.
   - Remove `mod media_runtime` and all `pub use media_runtime::{...}` exports
     from `crates/puffer-core/lib.rs`.
   - Add a `puffer-media` dependency in `crates/puffer-core/Cargo.toml`.
   - Import `ExactMediaDiscoveryCache` in `state.rs` and
     `runtime/internal_tool_permissions.rs` from `puffer_media`.
   - Update Bash/tool dispatch fallback calls from
     `crate::ExactMediaDiscoveryCache::empty` to `puffer_media`.
   - Update image/video workflow tools to import exact runtime APIs and the
     image count validator from `puffer_media`.
   - Keep workflow tests focused on request/config/permission wiring. Media
     adapter behavior belongs in `puffer-media` tests.

5. Update `puffer-cli`.
   - Add a `puffer-media` dependency in `crates/puffer-cli/Cargo.toml`.
   - Update `daemon.rs` and `desktop_api.rs` imports for exact media cache,
     media generation requests, preview helpers, timeline attachment helpers,
     and generated video access metadata.
   - Keep daemon DTOs and generated video ticket serving in `puffer-cli`.

6. Keep resources and tools in place.
   - Do not move provider YAML or internal tool resources.
   - Keep `puffer-resources` tests responsible for YAML/schema/skill contracts.
   - Keep `puffer-tools` tests responsible for internal-only registry behavior.

7. Verification.
   - Run `cargo test -p puffer-media`.
   - Run core adapter filters:
     `cargo test -p puffer-core --lib image_generation`,
     `cargo test -p puffer-core --lib video_generation`, and
     `cargo test -p puffer-core --lib internal_tool_permissions`.
   - Run CLI media filters:
     `cargo test -p puffer-cli media` and
     `cargo test -p puffer-cli generated_media`.
   - Run resource/tool contract tests:
     `cargo test -p puffer-resources --test image_catalog_governance`,
     `cargo test -p puffer-resources --test video_tool_schema`,
     `cargo test -p puffer-resources --test media_generation_skills`, and
     `cargo test -p puffer-tools internal_tool`.
   - Run one final `cargo test --workspace` for the migration PR only. Normal
     media feature work after this split should not require the full workspace
     command unless it changes cross-crate contracts.

## Risks And Mitigations

- Import churn can hide behavior changes. Mitigation: move files first, then
  make the smallest import and visibility edits needed to compile.
- Test fixtures can break because of relative `include_str!` paths. Mitigation:
  move fixture files with the adapter modules and keep relative paths stable.
- `puffer-resources` can leak into production dependencies through tests.
  Mitigation: put it only under dev-dependencies or replace the parser usage
  with direct `ProviderDescriptor` fixtures.
- `puffer-core` workflow tests may reach into media internals. Mitigation:
  rewrite them around public `puffer_media` facade behavior or keep that
  behavior tested inside `puffer-media`.
- Removing `puffer-core` re-exports will break consumers at compile time.
  Mitigation: update all in-repo imports in the migration and intentionally do
  not add compatibility shims.
- The new crate can become a dumping ground. Mitigation: keep workflow,
  permissions, daemon RPCs, resources, and tool registry outside it.

## Acceptance Criteria

- `crates/puffer-media` exists and is a workspace member.
- `puffer-media` production dependencies do not include app/runtime crates.
- `puffer-core` no longer owns `media_runtime*.rs` or `runtime/media/**`.
- `puffer-core` does not re-export media APIs for compatibility.
- In-repo consumers import media APIs from `puffer_media`.
- `cargo test -p puffer-media` covers media adapter, resolver, sidecar,
  preview, generated video, and Bash output parsing behavior.
- Focused core/cli/resources/tools commands above pass.
- A final full workspace test passes once for the migration.
