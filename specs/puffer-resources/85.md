# BytePlus Image Catalog Governance

## Summary

BytePlus image resources now distinguish native ModelArk model IDs from router
or gateway aliases and expose the current Seedream static catalog that fits the
existing `images_json` adapter.

## BytePlus Catalog

- `seedream-5-0-260128` is displayed as `Seedream 5.0 Lite`.
- `seedream-4-5-251128` is displayed as `Seedream 4.5`.
- Both models use provider-level `images_json` execution at
  `/images/generations`.
- Both models keep the documented select parameters used by the current
  adapter: `size=2K` and `output_format=png|jpeg`.

## Exclusions

- `seedream-5-0-lite-260128` and `dola-seedream-5-0-lite-260128` are not added
  because they are not confirmed as callable BytePlus image-generation model
  IDs for the current adapter.
- `seedream-4-0-250828` and `seedream-3-0-t2i-250415` are not added because
  deprecated or deactivated models are excluded.
- `bytedance/seedream-*` IDs remain Vercel AI Gateway aliases and are not valid
  BytePlus native resource IDs.

## Governance

A new `puffer-resources` integration test file covers image catalog governance
without expanding `src/model.rs`, which is close to the repository's 1000-line
Rust file limit.

The tests assert:

- bundled image media descriptors validate through the registry schema;
- image model IDs are unique per provider;
- native providers do not contain slash-style gateway aliases;
- Vercel image-only static fallbacks override provider-level chat image output
  with `images_json`;
- OpenRouter remains discovery-driven without static image fallbacks;
- BytePlus includes only the current native Seedream allowlist and excludes
  known aliases or deactivated IDs.

## Runtime Impact

Runtime and desktop capability discovery are unchanged. BytePlus remains a
static image provider, so startup performance and media capability resolution do
not gain network calls or provider-specific discovery logic.
