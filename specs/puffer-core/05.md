# Typed Session Permissions

## Purpose

Collapse `puffer-core` session permissions onto one canonical typed model
while preserving legacy-compatible session resume behavior where the historical
snapshot projection can express it.

## Design

`AppState::session_permission_state` is the only in-memory session permission
state in `puffer-core`.

`SessionPermissionState` carries:

- a session-wide allow-all bit
- typed session grants stored in `SessionPermissionGrants`

`EffectivePermissionProfile` is constructed from:

- cwd and working dirs
- workspace permission config
- workspace sandbox config
- current session id
- typed `SessionPermissionState`
- plan-mode context
- optional request-scoped tool filter

`DerivedPermissionPolicy` remains the executor-facing projection, but it is now
strictly typed. It exposes typed filesystem/process/network policy only. It no
longer supports any legacy executor bridge projection.

## Logic

Session permission mutation is typed-only:

- `allow_permission_for_tool_call()` records typed grants for the approved tool
  call
- `grant_all_tools_for_session()` flips the typed allow-all bit
- `replace_session_permission_state()` swaps typed state without maintaining
  any compatibility mirrors

Browser session approvals remain typed and category-scoped:

- Browser grants record Browser surface/category permissions
- Browser grants remain narrower than a whole-tool allow
- cross-session Browser access still requires an explicit cross-session grant

Runtime permission evaluation reads typed state directly:

- `load_runtime_permission_context_with_inputs()` passes
  `state.session_permission_state()` into `EffectivePermissionProfile`
- no runtime ingress path rebuilds typed state from legacy fields
- no fallback helper reconstructs current-turn permission state

Session store behavior remains typed-first with compatibility projection:

- `StateSnapshot` still does not serialize full typed permission grants
- resume may restore the legacy-compatible subset of session approvals through
  a compatibility projection
- typed-only narrow grants remain runtime-only until durable session schema
  expands
- snapshot persistence projects typed session state into:
  `session_allow_all: bool` plus `session_tool_permissions: map<string, string>`
- resume rebuilds typed session state from those optional compatibility fields
- older snapshots missing those fields still replay as the empty legacy
  projection

## Contracts

- `SessionPermissionState` is the only session permission source of truth in
  `puffer-core`.
- No `AppState` legacy permission mirrors exist.
- No session permission API in `puffer-core` accepts or emits legacy tool
  permission maps.
- No executor bridge exists from typed permission policy back to a legacy
  permission DTO.
- Resume restores the legacy-compatible subset of session approvals.
- Resume does not restore typed-only narrow grants that have no legacy
  projection.
- Browser session approvals are evaluated only through typed category/surface
  grants.
- Request-scoped deny/allow rules continue to apply on top of typed session
  state.

## Tests

Coverage should stay focused on typed behavior only:

- Browser typed grant/category behavior
- typed session allow-all behavior
- request-scope deny/allow behavior
- permission prompt mutations of typed session state
- worker/UI round-trips preserving typed session permission state
- session restore preserving only the legacy-compatible subset of typed
  permission state
- session resume restoring allow-all from `StateSnapshot`
- session resume restoring whole-tool approvals from `StateSnapshot`
- session resume still dropping Browser typed category grants
- derived typed executor policy preserving sandbox/approval fields
