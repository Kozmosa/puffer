# Runtime Message Actor Propagation

## Purpose

Runtime state tracks the current message actor so root agents and spawned
subagents can emit distinguishable transcript, stream, and workflow messages.

## Design

`AppState` stores an optional current actor. Root sessions synthesize an
`agent` actor from the session id, child sessions synthesize a `subagent` actor
from `parent_session_id`, and spawned agent-tool executions install a concrete
subagent actor with the generated agent id and agent type before the nested turn
starts.

User and system helpers synthesize user/system actors with the active session
and team context. Workflow mailbox writes persist the actor alongside stored
messages while preserving the existing root `from` routing string (`user` for
plain SendMessage, team lead for team-lead sends) and using the concrete actor
id for subagent/team-member sends.

## Contracts

Provider request construction remains unchanged; actor metadata is local runtime
metadata and is not sent as model conversation content. Actor fields are additive
and optional in durable stores so older transcripts and workflow mailbox files
continue to load.
