# puffer-core update 133

Verified Lambda Skill gates can now be backed by either a checked-in host
catalogue path or an external `lskillc export-json` compiler run.

## Design

`runtime::lambda_gate::gate_for_verified_skill` is the single entry point used by
skill side turns. For `lambda-skill` verification metadata it first reads an
explicit `host_catalogue_path`. If no path is present, it can compile the formal
`.lskill` source on demand when `PUFFER_LAMBDA_SKILL_GATE` is set to `compile`,
`strict`, `on`, `true`, `yes`, or `1`.

Compiler discovery is local-only and does not vendor skill libraries into
Puffer. This discovery contract has since been narrowed by update 52 to explicit
configuration only. At the time of this update it checked:

- `PUFFER_LSKILLC`
- `lskillc` on `PATH`

The default remains prompt-only for Lambda Skills without an explicit host
catalogue. This is intentional: the translated corpus currently uses formal
host-tool names that are not yet one-to-one Puffer runtime tool names, so
turning strict compilation on globally would make those skills fail closed until
the host-tool bridge is installed.

## Contracts

Strict compiler mode is fail-closed. If `PUFFER_LAMBDA_SKILL_GATE` asks for
compiled catalogues but the skill lacks `source_path`, the compiler cannot be
found, or `lskillc export-json` fails, the skill command fails before starting
the side turn.

The compiler output is parsed by the same typed gate as checked-in catalogues.
No generated host catalogue is written into the Puffer repository.

## Verification

`cargo test -p puffer-core runtime::lambda_gate` covers catalogue-file loading,
compiler discovery, and a fake external `lskillc export-json` binary in addition
to the gate parser and stateful call checks.
