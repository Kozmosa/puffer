# 16. External Path Approval Prompts

## Summary

Claude-compatible filesystem tools no longer fail immediately when a model
targets a path outside the current working directories. The runtime now asks
for manual approval before granting that path to the current tool call.

## Design

- `Read`, `Write`, `Edit`, `NotebookEdit`, `Glob`, and `Grep` inspect their
  path-bearing input fields before execution.
- If the target is already inside the effective filesystem policy roots, the
  tool runs unchanged.
- If the target is outside the roots, the runtime emits a permission prompt
  naming the tool and the path root it would grant.
- `Allow once` extends only the in-flight filesystem policy.
- `Allow session` stores a path-prefix grant in typed session permission state
  and reloads the effective filesystem policy.
- `Allow all for session` grants the path prefix and keeps the existing
  session-wide tool approval behavior.
- `Deny` returns a normal tool denial result instead of surfacing the old
  `/add-dir` error directly to the transcript.

## Compatibility

Existing `/add-dir` roots and workspace permission files keep their behavior.
Session path grants are append-only typed grants and are folded into the
effective workspace roots for subsequent tool calls.

## Tests

Runtime tool tests cover denying an external `Read`, allowing an external
`Read` once, and reusing an `Allow session` path grant across repeated `Glob`
permission resolution.
