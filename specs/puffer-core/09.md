# LSP Diagnostics Notification Drain

## Purpose

Persist diagnostics notifications that arrive just after an LSP request
response when a file has been opened or changed.

## Design

After `with_lsp_session` syncs a file with `didOpen` or `didChange` and
runs the requested operation, it now drains pending server messages until
the channel is idle for a short 50 ms window. Calls that do not sync a file
keep the previous immediate drain behavior.

## Logic

LSP servers publish diagnostics asynchronously. A server can legally return
the response to `textDocument/hover` before the reader thread has delivered
the preceding `textDocument/publishDiagnostics` notification to the client
loop. The immediate non-blocking drain could miss that notification, so the
diagnostics cache stayed empty until a later operation. The idle drain
handles any server requests and notifications that arrive in the small
post-operation window, then returns once no more messages are available.

## Contracts

- Request handling still waits for the normal response using the existing
  request timeout.
- The idle wait only applies to operations that synced a file and is capped
  by the short idle timeout, so workspace-symbol and diagnostics-only calls
  retain their previous latency profile.
- Existing publish-diagnostics parsing and storage format are unchanged.

## Compatibility

This preserves LSP protocol behavior while making diagnostics visible to
the next user-facing diagnostics query more reliably.
