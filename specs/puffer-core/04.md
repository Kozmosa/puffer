# Skill and MCP server hot-reload

## Purpose

Hot-reload of skills and MCP servers — a long-running puffer session (TUI,
remote managed agent, or otherwise) picks up newly added, modified, or
removed skill and MCP-server resource files without restarting the process.

## Design

Three layers cooperate:

1. **Hot-swappable runner roster** — `LocalToolRunner.mcp_host` is wrapped
   in `Arc<RwLock<McpHost>>`. The MCP-facing `ToolRunner` impl uses a read
   lock on every call; the resource reload path takes a write lock to
   install a fresh `McpHost`. The previous host is dropped, which
   asynchronously terminates its `kill_on_drop` MCP child processes; the
   connection manager lazily spawns whatever the new roster needs on the
   next call.

2. **Downcast seam on the trait** — `ToolRunner` gains an
   `as_any(&self) -> Option<&dyn Any>` method (default `None`).
   `LocalToolRunner` returns `Some(self)` so
   `puffer_core::reload_runtime_resources` can reach the inherent
   `LocalToolRunner::replace_mcp_roster(servers, workspace)` method
   without serializing or otherwise crossing the trait boundary. Remote
   runners (`puffer-runner-grpc`) leave `as_any` as `None` because they
   own MCP lifecycle out of band.

3. **Filesystem watcher** —
   `puffer_core::runtime::resource_watcher::ResourceWatcher` runs a
   `notify::RecommendedWatcher` over the builtin, user, and workspace
   resource roots (top-level, recursive). Events are filtered against a
   known set of resource subdirectories
   (`skills`, `mcp_servers`, `mcp`, `plugins`, `prompts`, `agents`,
   `tools`, `hooks`, `ides`, `mascots`, `providers`) and debounced into a
   single shared `Arc<AtomicBool>` signal carried on `AppState` as
   `resource_reload_signal`.

## Logic

`AppState::take_reload_request()` ORs the in-loop
`reload_resources_requested` flag with the cross-thread signal, returning
true once and clearing both. The TUI loop calls
`maybe_apply_requested_reload` at the top of each iteration, **but only
when no turn is in flight** (`tui.has_pending_submit()` is false). This
matters because `reload_runtime_resources` calls
`shutdown_runtime_services()` (sleeps 500 ms, waits up to 30 s for
background agents to drain, kills LSP) and hot-swaps the live MCP host;
running that mid-turn would freeze the render loop and orphan in-flight
MCP child processes. The latched signal stays set across the busy
window — the next idle tick picks it up.

Reload errors from `load_resources` (e.g. a SKILL.md whose YAML
frontmatter is mid-edit when the watcher fires) are caught inside
`maybe_apply_requested_reload` and emitted as a system message rather
than propagated. Without this, a single bad save would kill the TUI and
discard the in-memory transcript. Command-driven reloads
(`/reload-plugins`, `/mcp reload`) go through the same helper so they
get the same treatment.

`reload_runtime_resources` already swapped `*resources`, rebuilt the
provider registry, and shut down long-running services. It now also calls
`apply_mcp_roster_to_runner`, which downcasts `state.tool_runner` and (if
local) installs the new MCP roster derived from
`resources.mcp_servers + plugin_mcp_servers(resources)` — the same merge
puffer-runner-local performs at startup.

Skill content does not need explicit fan-out: the system prompt is rebuilt
from `resources.skills` on every turn, so the resource swap alone is
sufficient.

The watcher creates the user and workspace resource directories on first
start if they are absent so it can attach even on a fresh checkout; the
builtin root is attached only when it exists (we never write into the
install location). Watcher startup errors are non-fatal — the user can
still trigger reloads with `/reload-plugins` or `/mcp reload`.

Debounce window is 200 ms; signal flips inside the window are coalesced.
Editor swap files, dotfiles, and files outside the recognized resource
subdirectories are filtered out at the event level so unrelated writes
under `.puffer/resources/` (e.g. `.git/`, `.DS_Store`) don't trigger a
reload.

## Contracts

- `state.tool_runner: Arc<dyn ToolRunner>` retains the same trait surface;
  consumers that use only `Arc<dyn ToolRunner>` see no change.
- `LocalToolRunner` is still `Clone`. All clones share one
  `Arc<RwLock<McpHost>>` so a hot-swap from any clone is visible to every
  other live handle without a fresh `Arc<dyn ToolRunner>` allocation.
- `replace_mcp_roster` is inherent on `LocalToolRunner`; callers reach it
  via the `as_any` downcast. Adding a new resource type to `LoadedResources`
  also requires extending the merge inside `collect_runner_mcp_roster` (or
  the equivalent helper) and listing its directory in
  `resource_watcher::WATCHED_SUBDIRS`.
- The watcher signals reload *requests*; it does not perform the reload
  itself. The actual reload happens on the next iteration of the consumer
  (typically the TUI loop) under the existing `reload_runtime_resources`
  surface, preserving error reporting / system-message emission.
- `ResourceWatcher::start` may create `user_config_dir/resources` and
  `workspace_config_dir/resources` directories so it can attach; the
  builtin root is only attached when it already exists.

## Compatibility

- `ToolRunner` trait gained a default-implemented `as_any` method, so all
  existing implementors compile unchanged. Remote runners leave the
  default in place.
- The pre-existing `/reload-plugins`, `/mcp reload`, and `/mcp enable|disable`
  commands continue to work and now go through the same hot-swap path on
  the local runner.
- Tests: new
  `crates/puffer-core/tests/hot_reload.rs` covers the end-to-end
  add/modify/remove flows for skills and MCP servers plus the cross-thread
  signal handoff; new `runtime::resource_watcher` unit tests cover the
  debounce window, dotfile/swap-file filtering, and recognition of
  resource subdirs.
