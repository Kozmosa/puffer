# puffer-core update 06: /genskill runtime boundary

## Summary

`/genskill` runs the GEPA loop from synchronous slash-command dispatch while
non-interactive mode may already be executing inside a Tokio runtime. The
command must not start or block nested Tokio runtimes on active runtime worker
threads.

## Design

- `handle_genskill_command` still builds an execution trace from `AppState`,
  constructs a `PufferAgentRuntime`, runs `puffer-skill-evolution::run_gepa`,
  and writes the selected `SKILL.md` under `resources/skills/`.
- GEPA orchestration remains asynchronous internally and is bridged from the
  synchronous command handler with a small blocking helper.
- Each `PufferAgentRuntime::invoke_agent` call runs the synchronous
  `execute_user_prompt` provider path on a plain OS thread whenever the caller
  is already inside Tokio. This keeps nested runtime users, including MCP
  discovery and blocking provider adapters, outside active runtime worker
  threads.

## Compatibility

- Slash-command output and `/genskill --candidates/--rounds` parsing are
  unchanged.
- Existing provider dispatch, auth, resource loading, MCP discovery, and
  transcript mutation semantics are preserved.
- The change only affects the runtime boundary used by `/genskill` sub-agent
  calls and is intended to make interactive and non-interactive invocation
  behave the same.
