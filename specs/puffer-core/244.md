# puffer-core Update 244: Media Runtime And Capability Resolution

## Summary

Puffer should replace the current OpenAI-only `ImageGeneration` environment
variable path with a small media runtime contract used by desktop UI, slash
commands, and agent tools.

The runtime has four responsibilities:

- Resolve verified image and video capabilities from connected providers.
- Execute image and video generation through provider adapters.
- Track long-running media jobs.
- Persist generated media as artifacts instead of transcript base64.

The first implementation should not require a new workspace crate. Keep the
contracts stable enough to extract later if desktop and core implementations
start duplicating substantial provider logic.

## Research Basis

OpenAI supports direct image model calls through the Images API and
conversational image generation through the Responses API
`image_generation` tool. The distinction means chat model selection and media
model selection should remain separate. Reference:
https://developers.openai.com/api/docs/guides/image-generation.

fal and Replicate model production media generation as durable asynchronous
work. fal exposes queue states, status streaming, result retrieval, webhook
completion, and cancellation. Replicate exposes prediction lifecycle states and
requires callers to persist output files when long-term access is needed.
References:
https://fal.ai/docs/documentation/model-apis/inference/queue,
https://replicate.com/docs/topics/predictions/lifecycle, and
https://replicate.com/docs/topics/predictions/output-files.

## Architecture

### MediaCapabilityResolver

The resolver combines:

- connected provider credentials
- provider registry descriptors
- provider API discovery results
- deterministic provider adapter rules
- optional builtin hints
- optional agent-inferred candidate metadata

Only adapter-verified capabilities are executable. Agent-inferred metadata can
reduce manual maintenance by proposing model capability records, but the
runtime must treat it as non-executable until validation succeeds.

Provider YAML may carry optional media hints, but it is not the authority for
runtime execution. The resolver owns the final `available` decision.

### MediaProviderAdapter

Each provider adapter exposes a small interface:

```rust
trait MediaProviderAdapter {
    fn provider_id(&self) -> &str;
    fn capabilities(&self, auth: &ProviderAuth) -> Result<Vec<MediaCapability>>;
    fn submit_image(&self, request: ImageGenerationRequest) -> Result<MediaJobUpdate>;
    fn submit_video(&self, request: VideoGenerationRequest) -> Result<MediaJobUpdate>;
    fn poll_job(&self, job: &MediaJob) -> Result<MediaJobUpdate>;
    fn cancel_job(&self, job: &MediaJob) -> Result<MediaJobUpdate>;
}
```

Synchronous providers can return a completed job update immediately. Async
providers return a queued or running job with an external request id.

### MediaGenerationService

The service owns validation, request normalization, adapter selection, job
creation, artifact persistence, and event emission. It should be the only path
used by desktop media actions and the existing `ImageGeneration` tool.

In the first implementation, this can be a module boundary rather than a new
crate. A shared crate is justified only after both desktop and core need the
same concrete adapter code.

### MediaJobStore

Jobs are durable sidecar state, not transcript content. The initial backing
store should be JSON sidecars under existing Puffer session or workspace data
directories, not a new database:

```rust
struct MediaJob {
    id: String,
    session_id: Option<String>,
    provider_id: String,
    model_id: String,
    kind: MediaKind,
    operation: String,
    status: MediaJobStatus,
    prompt: String,
    settings_json: serde_json::Value,
    external_request_id: Option<String>,
    artifact_ids: Vec<String>,
    error: Option<String>,
    created_at_ms: u64,
    updated_at_ms: u64,
}
```

Core statuses should normalize provider-specific states:

- `queued`
- `running`
- `streaming`
- `completed`
- `failed`
- `canceling`
- `canceled`
- `expired`

### MediaArtifactStore

Artifacts store bytes and metadata outside the transcript:

```rust
struct MediaArtifact {
    id: String,
    job_id: String,
    kind: MediaKind,
    mime_type: String,
    local_path: String,
    external_url: Option<String>,
    external_url_expires_at_ms: Option<u64>,
    width: Option<u32>,
    height: Option<u32>,
    duration_ms: Option<u64>,
    provider_id: String,
    model_id: String,
    prompt: String,
    revised_prompt: Option<String>,
    created_at_ms: u64,
}
```

Image and video bytes are written under a session or workspace artifact
directory. Transcript events reference artifact ids or local paths. Remote
object storage is out of scope for the first implementation.

## Capability Contract

The provider registry may expose media hints, while the resolver exposes
verified media capabilities:

```rust
struct MediaCapability {
    provider_id: String,
    model_id: String,
    kind: MediaKind,
    operations: Vec<String>,
    async_job: bool,
    streaming: bool,
    parameters: serde_json::Value,
    status: CapabilityStatus,
    reason: Option<String>,
    source: CapabilitySource,
    checked_at_ms: u64,
}
```

`CapabilitySource::AgentInferred` is advisory only. Runtime execution requires
`CapabilityStatus::Available` from a deterministic adapter check or a provider
API response the adapter knows how to interpret.

The first resolver should support static adapter verification for known
providers and cached provider model discovery. It should not ask an LLM to
classify capabilities during normal user requests.

## Tool Integration

The existing `ImageGeneration` tool should become a thin caller into
`MediaGenerationService`:

- It resolves the configured image provider and model from media settings.
- It validates prompt length, output preferences, and connected provider auth.
- It creates a media job and artifact.
- It returns structured job and artifact metadata.

Video should be introduced as a separate tool only after the job and artifact
contracts are in place. Video generation should never block the main agent turn
for the full render duration unless the provider completes within a short
bounded wait.

## Runtime Behavior

- Explicit `/image` and `/video` prefixes are the first trigger path.
  Natural-language auto-detection and composer media modes are optional later
  work and must be conservative.
- Images may stream partial previews when the provider supports them.
- Videos are async job-first. Polling, status streaming, and cancellation are
  part of the core contract.
- Provider URLs are treated as temporary unless the adapter marks them durable.
  The runtime downloads outputs that should survive session reload.
- Cost, usage, provider request id, and revised prompt should be preserved when
  returned by a provider.

## Performance And Stability

- Capability discovery should be cached by provider, auth identity, media kind,
  and adapter version with a bounded TTL.
- Discovery failures should degrade to `unknown`, not block unrelated chat
  functionality.
- Large media files should be streamed to disk where provider APIs permit it.
- Base64 should be decoded at the backend boundary and not retained in
  transcript or long-lived in-memory structures.
- Job polling should use backoff and stop at terminal states.
- Cancellation should be idempotent from Puffer's perspective even when the
  provider cannot guarantee in-progress cancellation.

## Boundaries

- Do not build a general workflow engine for media generation.
- Do not let agent-inferred metadata bypass adapter validation.
- Do not store generated media inline in session transcript events.
- Do not couple media model selection to the chat `defaultModel`.
- Do not introduce remote object storage before local artifact storage needs it.
- Do not add a new shared crate until a concrete duplication problem appears.
- Do not implement natural-language trigger inference in the first pass.

## Brainstorming Review

The review removed or deferred the main overdesign risks:

- `puffer-media` as a new crate is deferred. Define stable DTOs and module
  boundaries first; extract only if desktop and core duplicate real adapter
  logic.
- Provider registry media fields are hints, not runtime truth. Adapter
  validation remains the authority.
- Job and artifact stores are logical stores backed by JSON sidecars and local
  files, not a database or object-storage abstraction.
- `/image` and `/video` are the first trigger path. Natural-language intent
  routing and composer media modes are future work.
- Video execution waits until the job/artifact lifecycle exists. The UI can show
  video settings earlier, but runtime support should not block image delivery.

## Execution Plan

1. Define media DTOs and settings structs for `MediaKind`, `MediaCapability`,
   `MediaJob`, `MediaArtifact`, image defaults, and video defaults. Keep these
   small and serializable.
2. Add `list_media_capabilities` behind the daemon boundary. The first pass
   should use connected auth plus deterministic adapters and return
   `unknown` or empty results on discovery failure.
3. Add the OpenAI image adapter path using media settings instead of
   `PUFFER_IMAGE_MODEL`. Decode base64 at the backend boundary and write the
   artifact to local storage.
4. Add JSON-backed media job and artifact sidecars. Normalize synchronous image
   completion into the same job lifecycle used later by async video.
5. Move the existing `ImageGeneration` tool onto `MediaGenerationService` and
   return job/artifact metadata. Keep the old direct OpenAI request path out of
   the tool after migration.
6. Add explicit `/image` submit handling. `/video` can create a validation error
   or queued job only after at least one video adapter exists.
7. Add a first video adapter after job polling, cancellation, and artifact
   persistence are tested. Prefer one queue-based provider before adding more.
8. Harden with resolver tests, artifact persistence tests, provider error tests,
   cancellation idempotency tests, and a regression test proving chat still
   works when media discovery fails.

## Verification

- Resolver tests should cover connected filtering, unavailable credentials,
  adapter-verified capabilities, agent-inferred advisory records, and cache
  invalidation.
- Service tests should cover sync image completion, async video submission,
  polling transitions, cancellation idempotency, provider errors, and artifact
  persistence.
- Tool tests should verify `ImageGeneration` uses media settings and returns
  artifact metadata rather than raw bytes.
- Regression tests should ensure chat turns still work when media capability
  discovery fails.
