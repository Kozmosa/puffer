# 122. Lambda Host Call Bridge

Puffer now has a formal-host-call bridge for strict Lambda Skill execution.

## Design

`LambdaHostCall` is a model-visible runtime tool used only while a Lambda Skill
gate is active. It accepts:

- `host_tool`: the formal host tool name from the verified catalogue
- `tool`: the concrete Puffer tool id that will implement the operation
- `input`: the exact JSON input for that concrete call

The bridge first checks `host_tool` against the active Lambda gate without
committing any registered facts. It rejects empty names and unavailable concrete
Puffer tools before recording state. If accepted, Puffer records a pending
bridge on `AppState`. The next concrete tool call must match both the declared
tool id and the declared JSON input exactly. Mismatches are rejected before
permission prompts or execution.

When the declared concrete call reaches the normal execution boundary after
Puffer permission and filesystem checks, Puffer commits the formal `host_tool`
step through the Lambda gate, clears the pending bridge, and then dispatches the
concrete tool.

## Contracts

The bridge keeps formal host-tool names separate from Puffer runtime tool names.
This avoids weakening the Lambda gate to effect-only checks and avoids requiring
the translated corpus to rename every formal catalogue tool to a Puffer tool id.

The concrete Puffer tool still goes through Puffer's normal permission,
filesystem, hook, and provider execution paths. The Lambda gate authorizes only
the formal host operation and its trace facts; the concrete tool remains governed
by Puffer's existing runtime policies.

## Verification

`cargo test -p puffer-core lambda_host_call_` covers successful formal-to-
concrete bridging, deferred fact commitment, exact-input matching, mismatch
rejection, and unavailable concrete-tool rejection.
