# Project ACL permissions and internal Browser gating

## Purpose

Replace sandbox-centered runtime decisions with a project ACL that is evaluated
for every tool call. The runtime now grants ordinary file access inside the
current working directory by default, asks for access outside that directory,
gates shell commands by effective command name, and gates Browser activity by
current site and action.

## Design

`permissions/acl.rs` owns the project ACL format, parser, writer, and evaluators.
The ACL is stored in `.puffer/permissions.acl` under the project workdir and is
loaded into `RuntimePermissionContext` with the legacy permissions file and
session permission grants.

Supported ACL lines are scoped by a workdir header:

- `[project workdir]`
- `<priority> Allow all`
- `<priority> Deny all`
- `<priority> Allow|Deny read file|dir <path>`
- `<priority> Allow|Deny write file|dir <path>`
- `<priority> Allow|Deny bash command <cmd>`
- `<priority> Allow|Deny browser domain <domain>`
- `<priority> Allow|Deny browser action <action> <domain>`

Higher priority wins. If priorities tie, later ACL lines win.

## Logic

Filesystem access is allowed for paths under the current working directory.
Paths outside the current working directory ask unless an ACL rule allows them.
ACL denies are checked before the default cwd allow. Session yolo mode still
sets `allow_all_tools`, which bypasses permission prompts.

`AllowOnce` decisions are transient and are not written to the ACL. Session
"always allow" decisions append project ACL rules. File requests write read or
write path rules; shell requests write `bash command` rules; Browser requests write
domain/action rules; allow-all writes `Allow all`.

Bash and PowerShell calls are normalized to an effective command name. Shell
wrappers such as `bash -c`, `sh -lc`, `zsh -c`, `fish -c`, `pwsh -Command`, and
`cmd /c` are unwrapped so `bash -c "ls"` evaluates as `ls`. Lightweight wrappers
such as `env`, `command`, and `exec` are peeled before evaluation. Compound
shell strings with control or redirection operators ask unless explicitly
handled by the caller because one visible command name is not enough to describe
the behavior.

Browser calls are evaluated against the normalized Browser permission context.
The context includes the action set, target URL, current tab URL when available,
origin, host, and registrable domain. Approval for `google.com` does not approve
actions after a redirect to `baidu.com`; the current target domain is evaluated
again on each call.

## Compatibility

Legacy `permissions.toml` and browser policy code remain loadable while the ACL
becomes the durable project approval surface. Compatibility structs that still
carry sandbox names do not define filesystem breadth; project ACL, cwd defaults,
and yolo grants do.

`/sandbox` now reports that sandbox mode has been removed and points users to
`/permissions`.

## Verification

- ACL parser, path, Bash, and Browser evaluator unit tests.
- Runtime permission tests for Bash, Browser, filesystem grants, and yolo mode.
- Tool execution tests for persisted approval behavior.
