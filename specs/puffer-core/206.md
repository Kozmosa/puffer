# Canvas — agent-composed result pages (a design system, not a schema)

## Summary

The `Canvas` tool lets the agent render a structured result as an interactive
page instead of a long text answer. Canvas is a small **design system for the
agent**, not a fixed template: the agent **composes** a tree of documented
components for the task at hand (code review, research dashboard, test report),
and a shared renderer projects that tree to a self-contained HTML page (dark +
light) written under `.puffer/canvas/` and opened in the browser.

This is deliberately *not* "agent writes free HTML" (inconsistent, unsafe) and
*not* "agent fills one rigid schema" (a report DSL). It is a typed component
composition guided by a machine-readable component catalog and task recipes.

## Design

### Layers (atomic-design for the agent)
- **Atoms** — design tokens (color/space/type/radius), baked into the renderer's
  CSS, dark + light. The agent never reinvents colors or layout.
- **Components** — composable primitives rendered by a registry (one function per
  type): layout (`section`/`grid`/`columns`/`card`/`divider`), text
  (`heading`/`text`/`badge`), data (`metrics`/`kv`/`table`), viz (`bar`/`heatmap`),
  evidence/rich (`finding`/`diff`/`code`/`callout`).
- **Catalog** — `resources/canvas/components.md`: machine-readable usage knowledge
  (purpose / use_when / avoid_when / props per component). This is what makes
  Canvas a *design system* rather than a schema — it teaches the agent how each
  component *should* be used, not just that it exists.
- **Recipes** — task-level composition rules as skills, e.g.
  `resources/skills/canvas-code-review/SKILL.md`.

### Emission & transport
- Tool `Canvas` (`resources/tools/canvas.yaml`, handler
  `runtime:workflow:canvas`). Input is the component tree:
  `{ title, meta?, summary?, body: Node[] }`, `Node = { type, ...props, children? }`.
- Handler (`crates/puffer-core/runtime/claude_tools/workflow/canvas.rs`) renders
  the tree into a self-contained HTML document (template embeds the registry +
  tokens), writes it under `<cwd>/.puffer/canvas/canvas-<ts>.html`, opens it
  best-effort, and returns `{status, path, nodes, opened}`.

### Anti-slop discipline (structural, not stylistic)
The catalog and the code-review recipe require that claims carry **evidence** and
**actions**: a `finding` should cite `locations` and show `evidence` (the actual
diff), and expose `actions` (fix/test/explain) when the user can act. A node is
useful because it lets the reader inspect or act — not because it is decorative.
Decorative cards/metrics/charts untied to evidence are discouraged by the recipe.

### Action round-trip (the point, not a follow-up)
Canvas is an *entry to the next action*, not a static report. Each `finding`
action is a button that **continues the conversation without typing**: when a
daemon is reachable, the handler embeds a bridge `{ url, token, sessionId }` from
the daemon handshake into the page; clicking an action opens a WebSocket to the
daemon and calls `run_agent_turn` with the action's **bundled context** (the
finding title, severity, locations, and the diff evidence + an intent-specific
instruction). The agent's response continues in Corbina/TUI. With no daemon (CLI),
the action degrades to copying the bundled message to the clipboard.

Security note: embedding the daemon token in an on-disk HTML is a deliberate
tradeoff for the "open a file in the browser" model; the canvas file is written
owner-only (0600). Corbina-native embedding (no token on disk) is the hardened
follow-up.

### Renderer robustness
Models emit varied node shapes. The renderer coerces defensively: any
non-string body/evidence is reduced to text (objects with `code`/`diff`/`text`/
`value`/`content` fields are unwrapped), locations accept `path|file|location`
with optional `line`, and actions accept `intent` or `kind`. The page never
shows `[object Object]` or `undefined:NN`.

## Tradeoffs / what this is not (yet)

Reviewed adversarially (with a second model). The known limitation: the component
set is a flat namespace mixing layout/content/evidence, which can drift toward
"attractive report widgets". The north-star fix (documented follow-up, not in
this change) is a typed **Canvas IR** with semantic wrappers (`Issue`/`Change`/
`Risk`/`Task`) over evidence (`FileRef`/`DiffView`/`CommandOutput`) and
first-class actions (`PatchAction`/`OpenFileAction`), plus provenance validation
per recipe, with HTML/TUI/markdown as projections. This change ships the working
HTML projection + the catalog + a recipe + the node-action round-trip (via the
daemon bridge). The typed IR / semantic-wrapper layering remains the documented
follow-on.

## Testing
- `cargo test -p puffer-core canvas::` — renderer embeds title/spec/registry;
  `execute_canvas` writes a valid HTML file and counts body nodes.
- Manual: rendered both a code-review and a research dashboard from the same
  component set, confirming composition across task types (not task-locked).
