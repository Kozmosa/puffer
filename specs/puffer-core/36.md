# Internal Tool Permission Protocol

## Purpose

Remove Browser permission inference from shell command text. Browser actions
launched through first-party internal tools now ask permission through a
structured localhost callback owned by the running Bash tool invocation.

## Design

`runtime/internal_tool_permissions.rs` resolves authenticated internal requests
against the same Browser permission context used by direct Browser tool calls.
The resolver accepts `tool_id = "browser"` with Browser-native JSON input,
hydrates current-tab context, evaluates ACL/session policy, and prompts through
the existing permission prompt path when policy returns `ask`.

`claude_tools/bash.rs` starts a nonblocking loopback broker for foreground and
interactive Bash executions. The child shell receives:

- `PUFFER_INTERNAL_PERMISSION_ADDR`
- `PUFFER_INTERNAL_PERMISSION_TOKEN`
- `PUFFER_INTERNAL_PERMISSION_REQUIRED=1`

Internal tools send one JSON line containing protocol version, token, tool id,
and structured input. The broker validates the version and token, delegates to
the resolver, and returns one JSON decision line.

Background Bash cannot hold the mutable runtime permission resolver after the
tool call returns, so it exports only `PUFFER_INTERNAL_PERMISSION_REQUIRED=1`.
Internal tools launched from background Bash fail closed instead of running
without Browser permission.

## Logic

Browser permission state is no longer derived from `Bash.command`.
Shell commands are evaluated only by the shell command policy. Browser grants
are created and reused only for direct Browser tool calls and structured
internal Browser permission requests.

Session Browser approvals from either route share the same grant model:

- `allow_once` allows only the current action.
- `allow_session` records the suggested Browser grant scope and appends the
  matching Browser ACL rule.
- `allow_all_session` records the Browser grant and appends the project
  allow-all ACL rule.
- `deny` returns a denied internal-tool response and prevents the CLI action.

The broker reads accepted streams without blocking on partial writes. Requests
over the protocol size limit are denied and removed from the pending queue.

## Contracts

- No shell command parser is used for Browser permissions.
- Internal permission requests are structured JSON, authenticated by a per-Bash
  invocation token, and scoped to loopback.
- Direct Browser tool calls and internal Browser requests must produce
  compatible Browser permission contexts and session grants.
- `Bash` is not parallel-safe while it can host internal permission callbacks.
- A missing callback is allowed for human `puffer browser ...` usage, but
  hidden `puffer internal-tool browser ...` requires a callback.

## Verification

- Bash unit tests cover broker env export and token validation.
- Browser permission execution tests cover grant reuse in both directions
  between direct Browser tool calls and internal Browser requests.
- Runtime permission tests assert `Bash.command` text does not normalize into
  Browser context.
