# puffer-core Update 247: Preserve Provider Image Size Tokens

## Summary

`ImageGeneration` must preserve provider-specific image size tokens when a
chat turn includes an aspect hint. The configured media route is authoritative:

```text
provider_id + model_id + adapter + parameters
```

For BytePlus Seedream models, `size = 2K` is a valid provider parameter.
Replacing it with OpenAI-style dimensions such as `1024x1024` or `1024x1536`
makes the selected model fail local capability validation before the request is
sent.

## Runtime Contract

When the tool input includes `aspect`:

- if the selected settings contain `aspect_ratio`, map the aspect to that
  parameter;
- if the selected settings contain an OpenAI-style dimension `size`, map the
  aspect to the corresponding dimension size;
- if the selected settings contain a provider-specific `size` token such as
  `2K`, keep that token unchanged;
- if neither `aspect_ratio` nor `size` is configured, reject the aspect as
  unsupported.

The prompt can still carry composition guidance such as portrait or landscape.
The runtime must not invent unsupported parameter values to satisfy that hint.

## Compatibility

OpenAI-compatible image settings that use dimension sizes keep the existing
behavior. BytePlus and other exact image providers with enumerated resolution
tokens keep their selected default values and continue through capability
validation.

## Tests

The core image workflow has a regression test that builds a BytePlus-like
request with `size = 2K` and `aspect = portrait`, and verifies the generated
request preserves `2K`.
