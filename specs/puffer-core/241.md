# OpenAI Responses Incomplete Handling

## Summary

OpenAI Responses payloads with `status: incomplete` or streaming
`response.incomplete` events are treated as API-level failures instead of
successful terminal responses.

## Architecture

- The SSE parser raises a typed OpenAI stream API error for
  `response.incomplete`.
- The same typed error is reused for non-SSE JSON Responses payloads whose
  top-level `status` is `incomplete`.
- Non-SSE JSON Responses consumers, including the reflection judge single-call
  path and OpenAI-backed WebSearch helper, must reject incomplete payloads
  before extracting assistant text.
- Successful terminal completion requires a parsed `response.completed` payload.
- Incomplete Responses errors are classified as retryable at the sampling
  stream-restart layer, not at the lower HTTP transport retry layer.
- The legacy SSE path used by WebSocket fallback applies the same bounded
  stream-restart policy as the main Responses session path.
- WebSocket event parsing reuses the same incomplete typed error; the WebSocket
  retry classifier treats it as retryable instead of terminal success.
- WebSocket transport failures mark the current session for HTTP/SSE fallback
  so later turns do not repeatedly attempt a failing WebSocket transport.
- API-level incomplete responses do not mark the session fallback latch; they
  remain per-attempt/per-turn retry signals.

## Runtime Contract

- Partial deltas emitted before `response.incomplete` may still reach live UI
  consumers, but the turn fails with an error such as
  `Incomplete response returned, reason: content_filter`.
- Incomplete responses do not consume or advance Responses threading state.
- Incomplete responses are not classified as stream truncation, but they do
  enter the same bounded stream-restart loop as Codex `CodexErr::Stream`.
- Incomplete reasons are not filtered at retry classification time. Codex maps
  every `response.incomplete` reason to `ApiError::Stream`, and `Stream`
  errors are retryable until the configured stream retry budget is exhausted.
- If WebSocket negotiation fails before the first request, the fallback SSE
  request still restarts on `response.incomplete`.

## Compatibility

- `response.failed` retry classification is unchanged: rate-limit, server, and
  overload failures remain retryable.
- EOF before `response.completed` is still treated as a retryable stream
  truncation.
- Header-only EOF recovery is not used for terminal events, because missing
  `data:` lines can hide failure or incomplete details.
