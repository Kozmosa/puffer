# Connector Monitor Tasks

## Summary

`/monitor <connection>` creates an enabled connector workflow that triages
incoming messages into workspace-level monitor tasks.

## Behavior

- Each monitor owns a memory file under `.puffer/runtime/monitors/` and passes
  that path to its triage prompt.
- Telegram and Slack monitors also try to launch a background backfill agent so
  recent visible messages can become tasks immediately.
- Monitor `TaskCreate` calls are stored in `monitor_tasks.json` when metadata
  marks `_monitor` or names a monitor connection.
- `TaskList`, `TaskGet`, and `TaskUpdate` include monitor tasks alongside the
  current session task store.
- `/tasks` offers monitor-specific action prompts and ignore choices. Ignoring
  a task marks it completed, appends the ignore reason to monitor memory, and
  tries to launch a background learner agent for future filtering.
- The `/tasks` dashboard renders active monitor tasks with bracketed choices in
  the task row, such as `[Ignore]` and `[Draft reply]`, and hides monitor tasks
  after they are ignored.
- Telegram and Slack backfill prompts require connector lookup calls to pass
  the monitored connection slug so multi-account monitors inspect the intended
  account.

## Compatibility

The command is additive. Session-scoped task ids and stores remain unchanged,
and non-monitor task picker rows do not gain monitor action or ignore choices.
