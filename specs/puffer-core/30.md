# puffer-core update 08

## Purpose

Adds the `/genskill` slash command for generating reusable skill
documents from the current conversation transcript via the new
`puffer-skill-evolution` crate.

## Design

`/genskill` is registered in `supported_commands()` as
`CommandKind::Local`. It accepts optional flags `--candidates N`
(default 3) and `--rounds K` (default 2). The handler
`handle_genskill_command` lives in
`crates/puffer-core/command_helpers/genskill.rs`.

The handler:

1. Parses the optional flags into a `GepaOptions`.
2. Iterates `AppState::transcript`, mapping role + text + tool-call
   names + error flag into the `TranscriptStep` shape that
   `puffer_skill_evolution::extract_trace` accepts.
3. Refuses with a friendly hint if fewer than 5 tool calls have
   occurred — mirrors hermes-agent's "5+ tool calls" threshold for
   what counts as a non-trivial task.
4. Constructs a `PufferAgentRuntime` adapter that delegates each
   `invoke_agent(prompt)` call to puffer-core's
   `runtime::execute_user_prompt`, returning the assistant's text.
5. Builds a `tokio` current-thread runtime and `block_on`s
   `run_gepa(...)` with the default generate/judge/mutate prompt
   builders.
6. Writes the chosen `SkillCandidate` to
   `resources/skills/<name>/SKILL.md` (creating parent dirs); on name
   collision, suffixes with `-v2`, `-v3`, ...
7. Returns a user-facing message of the form
   `"Skill written to <path>"`.

`PufferAgentRuntime` is private to the handler module. It holds:

- `state: Mutex<AppState>` — cloned at handler entry so the trait's
  `&self` async method can access it without an upstream lock.
- `resources: LoadedResources` — also cloned.
- `providers: ProviderRegistry`, `auth_store: Mutex<AuthStore>` — both
  required by `execute_user_prompt`.

Mutex wrapping is required because the `AgentRuntime` trait has
`&self` (not `&mut self`) async methods.

## Logic

Trace bridging filters transcript events to `User`, `Assistant`, and
`ToolCall` roles, extracting text content and tool-call names. The
extractor (`puffer_skill_evolution::extract_trace`) drops empty turns
that have no tool calls and derives `task_summary` from the first
non-empty user turn.

The 5-tool-call threshold is a heuristic: it bounds the cost of running
GEPA on a session that hasn't accumulated enough signal to be worth
distilling into a skill. The hint message tells the user to retry
after a more substantive task.

## Contract

**Command surface:** `/genskill [--candidates N] [--rounds K]`. Both
flags default to GepaOptions defaults.

**Output:** Always returns a user-facing message. On success, that
message contains the absolute path to the new skill file. On failure
(too few tool calls, generation failure, write error), the message
explains the cause.

**Resource output:** `resources/skills/<name>/SKILL.md` where `<name>`
is taken from the chosen candidate's frontmatter, with `-v2`, `-v3`
suffixes appended on collision. Parent directories are created
automatically.

**Resource templates:** `resources/skills/genskill/{SKILL,generate,
judge,mutate}.md` are bundled prompt templates. The current handler
uses `puffer-skill-evolution`'s `Default*Prompt` builders (inline
templates). A future update may swap the resources in for full
templating; the bundled markdown files exist now so `/skill:genskill`
also resolves to a usable fallback prompt.

**Determinism:** Generation uses parallel sub-agent invocations whose
outputs depend on provider sampling. Two `/genskill` runs on the same
transcript will not produce identical skills.

## Architecture Details

The handler is synchronous at the puffer-core boundary
(`handle_genskill_command` returns `Result<String>`, not a future) but
calls `run_gepa` (which is async) by building a current-thread
`tokio::runtime::Runtime` inside the handler and `block_on`-ing the
GEPA call. This matches how other puffer-core handlers bridge to async
puffer-tools.

No new runtime infrastructure is added. Sub-agent execution reuses
puffer-core's existing `runtime::execute_user_prompt` path; the
`PufferAgentRuntime` adapter only narrows the signature to the trait's
`(&self, &str) -> Result<String>` shape.

The handler does not modify `puffer-tools`, the `runtime:agent` tool,
or any provider/transport code. It is a pure additive change to the
local-command surface.
