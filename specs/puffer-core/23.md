# 21. Browser Daemon Handshake Fallback

## Summary

Browser tool execution now treats daemon handshakes as ordered candidates
instead of a single selected endpoint.

## Behavior

- A matching workspace `.puffer/daemon.handshake` remains the first candidate.
- The user-level daemon handshake remains a global fallback candidate.
- If the workspace candidate is stale or unreachable, Browser tool execution
  retries the same `browser_agent` request against the user-level daemon.
- If every candidate fails, the final connection or daemon error is returned.

## Compatibility

The Browser tool schema, permission model, and daemon protocol are unchanged.
The change only affects local discovery resilience when a stale workspace
handshake coexists with a live desktop daemon handshake.

## Tests

`puffer-core` Browser local-tool tests cover candidate ordering with both a
matching workspace handshake and a user-level fallback.
