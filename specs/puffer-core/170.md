# Lambda Result Refinement Checks

## Summary

Verified Lambda Skill execution now validates concrete host-call results before
the Lambda gate commits any facts derived from those results.

## Runtime Contract

- A formal host call still must be admitted before its concrete Puffer tool call
  can run, and the next concrete call must exactly match the admitted concrete
  tool name and input.
- After the concrete call succeeds, Puffer checks the formal host result type
  against the concrete result value before registering `registers` facts or
  result-refinement facts.
- Executable result refinements include primitive types, arrays, success values
  of `Result<Ok, Err>`, record fields, equality and numeric comparisons,
  supported semantic predicates, and string shape predicates such as
  `ends_with_pdf`.
- Positive custom result facts such as `report_saved(p)` are treated as fact
  producers after the executable parts of the result type pass. They are not
  accepted as semantic proof obligations unless Puffer has a concrete checker
  for that predicate.

## Compatibility

Unsupported executable result refinements fail closed at runtime, which prevents
the model from gaining proof facts from an unverified concrete result. The
pending host-call bridge is split into its own module so the Lambda gate remains
small enough to keep the runtime code reviewable.
