# Skill Action Obligation Runtime Guard

Skills with `requires_action: true` cannot complete the next no-tool response
as ordinary final text.

## Runtime Contract

- The guard observes completed `ToolInvocation`s, not raw model tool requests.
- A successful `Skill` invocation activates the guard only when the resolved
  skill resource has `requires_action: true`.
- The guard uses the same `Skill` input parser as the executor, including `/`
  and `/skill:` stripping.
- Failed `Skill` invocations do not activate or satisfy the guard.
- Any non-`Skill` invocation satisfies a pending obligation, regardless of
  tool success.
- When a batch contains `Skill` followed by a non-`Skill` tool, the obligation
  is satisfied before the next model call.

## No-Tool Semantics

The first no-tool response after activation is not final. The loop appends the
assistant response items, appends a user reminder, and asks the model to start
tool-driven work.

The second consecutive no-tool response fails plainly with assistant text that
starts with `No work was started`. The guard then clears pending state.

If the loop exhausts `max_turns` while an obligation is still pending, it returns
the same `No work was started` failure text instead of the last assistant text.

## Hook Points

Streaming and blocking loops own one `SkillActionObligation` per turn run. They
observe tool batches immediately after execution and consult the guard before
ordinary finalization hooks on no-tool responses.

## Non-Goals

- No provider `tool_choice=required`.
- No workflow or phase schema.
- No skill-name-specific runtime path.
