# MCP Local Tool Test Runner Isolation

## Purpose

Keep `runtime:list_mcp_resources` and `runtime:read_mcp_resource` unit
tests deterministic without requiring a real external `docs-server`
binary on the developer machine.

## Design

The local-tools unit tests use a small fake `ToolRunner` for the `docs`
MCP server cases. The fake returns typed MCP resource DTOs through the
same `ToolRunner` methods production uses, while avoiding subprocess
startup. Filesystem MCP tests continue to use `LocalToolRunner` because
they exercise the built-in live workspace resource path.

## Logic

`runtime/local_tools.rs` is responsible for converting runner MCP DTOs
into the legacy JSON payloads expected by model-visible local tools. It
does not need to validate stdio MCP process launch behavior; that path is
covered by the runner MCP integration tests. The fake runner gives the
serializer a stable docs resource and mirrors the normal unknown-server
error text so list/read error mapping remains covered.

## Contracts

- Production MCP subprocess routing remains unchanged.
- The fake runner implements the complete `ToolRunner` trait so tests still
  pass through the same local-tool functions as production dispatch.
- Unknown server errors still surface as plain messages that include the
  requested server and available docs server.

## Compatibility

No runtime behavior or external MCP manifest format changes. This is a
test-only fix that removes an accidental dependency on a non-existent
`docs-server` command.
