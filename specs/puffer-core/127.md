# puffer-core update 127

Lambda Skill host-call trace metadata is retained through tool execution.

## Design

`ToolInvocation` now carries structured metadata from `ToolOutput`. Lambda
Skill gate admissions and committed host calls write a `lambda_skill` metadata
object that records the event, formal host tool, concrete bridged tool when
present, concrete input for admission, and registered facts for committed calls.

## Architecture

The Lambda gate module owns metadata construction and merge behavior. Tool
execution asks the gate for metadata only after permission and filesystem checks
pass, then merges it into the concrete tool result before hooks and turn events
observe the result. Provider runtimes propagate tool metadata into their
`ToolInvocation` values.

## Contracts

- `LambdaHostCall` emits `host_call_admitted` metadata without committing gate
  facts.
- The next matching concrete tool emits `host_call_committed` metadata and
  includes any facts registered by the formal host signature.
- Direct host calls under an active gate also emit `host_call_committed`
  metadata.
- Non-Lambda tools keep `metadata = null` unless their runtime result already
  supplied metadata.

## Compatibility

The metadata is internal trace data. It is not sent back to model providers as
tool output text and does not change permission decisions, tool schemas, or
existing non-Lambda behavior.
