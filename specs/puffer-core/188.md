# GraphQL Result Refinements

## Context

Translated Lambda skills use result refinements such as `gql_success(r)` and
`graphql_errors_checked(r)` to say that a GraphQL response was not merely
produced, but checked for API-level errors. Treating those predicates as custom
fact producers is too weak for effectful API calls.

## Design

Puffer now recognizes both predicates as semantic runtime predicates. The
checker accepts JSON objects, JSON arrays, and strings containing JSON. It
rejects non-JSON strings and values that contain a non-empty `errors` or
`userErrors` field anywhere in the response tree.

## Runtime Contract

Result checking now evaluates these predicates before falling back to custom
fact-producer handling. A concrete call returning a GraphQL error payload will
therefore fail the Lambda host-call commit step for result types like
`Result<GqlResponse{gql_success(r)}, ShopifyErr>`.

## Compatibility

This is intentionally stricter. Successful GraphQL responses with absent,
empty, null, or false error fields continue to pass. Responses with populated
GraphQL `errors` or mutation `userErrors` fail closed.
