# Lambda Skill Concrete Input Contracts

Verified Lambda host catalogues may bind a formal host tool to concrete Puffer
tools through `concreteTools`. Runtime admission now also requires a
`concreteInputContracts` entry for every declared concrete tool binding.

Concrete input contracts are exact JSON patterns. `{"$arg":"name"}` requires the
concrete value to equal the formal host argument named `name`. `{"$template":"..."}`
renders a string from formal arguments before comparison. Object and array
patterns require the same shape and do not allow extra concrete fields.

`LambdaHostCall` admission checks the formal host call, formal argument types,
concrete tool binding, and concrete input contract before installing the pending
bridge. The pending bridge still requires the next concrete tool call to match
exactly and commits the host call only after that concrete tool succeeds.

Runtime host catalogue validation fails closed when a declared concrete tool
lacks an input contract, when the contract does not reference exactly the formal
parameter set, or when a parameter/result refinement uses a predicate the runtime
cannot evaluate. The runtime-supported refinement set includes numeric and
cross-argument comparisons, suffix predicates such as `ends_with_pdf`,
`valid_arxiv_id`, and `parsed_ok`; unsupported predicates remain rejected at
preflight and at the host-call boundary.
