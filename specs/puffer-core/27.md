# 27. Responses Include Fallback

## Summary

OpenAI Responses turns omit the encrypted reasoning `include` selector on the
first request, and still retry once without reasoning if a provider rejects an
opted-in selector.

## Design

- The Responses request body still sends a `reasoning` object for models that
  support reasoning, so thinking effort is preserved.
- `include: ["reasoning.encryptedcontent"]` is disabled by default because
  OpenAI and Codex-compatible endpoints have rejected that selector before the
  model can answer.
- Operators can opt back into the selector with
  `PUFFER_OPENAI_INCLUDE_REASONING_ENCRYPTED_CONTENT=1` when testing an endpoint
  that accepts encrypted reasoning content.
- The OpenAI Responses session classifies `include[0]` validation failures from
  the provider error text.
- Streaming and blocking Responses turns rebuild the same request with
  `supports_reasoning=false`, which removes both `include` reasoning selectors
  and the `reasoning` request object.
- The session disables reasoning for the rest of that prompt after the fallback
  so subsequent tool-loop requests do not repeat a failing opt-in selector.

## Compatibility

The fallback only applies to provider-side include validation failures. Auth,
model, quota, tool, and other request errors still surface unchanged.

## Tests

Rust coverage checks the include-error classifier, the default request body
without an encrypted reasoning include, and the explicit opt-in path.
