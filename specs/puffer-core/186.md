# 186. Secret Handle Modal Bridge

## Summary

Puffer adds two workflow tools for verified Lambda Skill contracts that need to
create Modal secrets without pretending a raw string is a safe concrete tool
input.

## Runtime

- `SecretValue` supports `action: "prepare"`.
- The prepared value is stored only in the current `AppState` under a generated
  `secretId`.
- The tool result is a handle object with `sec: "secret"` and never includes the
  raw value.
- `ModalAction` supports `action: "secretCreate"`.
- `ModalAction.secretCreate` accepts only a `SecretValue` handle, resolves it in
  the session store, validates that the stored value is a `KEY=VALUE`
  assignment, then runs `modal secret create <name> <assignment>` without a
  shell.
- Modal command failures redact both the full assignment and the secret value
  before returning stderr/stdout text.

## Lambda Gate Metadata

Tool schemas may mark input fields with `x-puffer-secret: true`, or define
top-level `x-puffer-secret-paths`. LambdaHostCall gate metadata redacts those
paths in both formal host args and concrete input before appending gate events.
The raw values remain available to the pending bridge for exact input matching
and to the concrete tool for execution; gate metadata uses the redacted copies.

## Compatibility

The bridge is intentionally fail-closed:

- Raw strings passed to `ModalAction.value` are rejected.
- Unknown or malformed secret handles are rejected.
- Invalid Modal secret names or non-`KEY=VALUE` assignments are rejected before
  any CLI process is spawned.
