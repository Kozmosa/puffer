# Typed-only runner gRPC transport

## Purpose

Define the final filesystem-policy transport contract for
`puffer-runner-grpc`.

## Design

`proto::ToolRequest.filesystem` is the only filesystem-policy field on the
wire.

`proto::RunnerCapabilities` reports backend identity, version, supported
tools, and MCP support only. It does not advertise a separate typed-filesystem
capability flag.

Removed proto field numbers stay reserved:

- `ToolRequest = 4`
- `RunnerCapabilities = 5`

## Logic

Encoding always populates `filesystem`.

Decoding requires `filesystem` to be present. If it is missing, the server-side
adapter returns `RunnerError::InvalidArgument("filesystem: missing")`, which
the gRPC surface exposes as `INVALID_ARGUMENT`.

Filesystem sandbox enum values round-trip exactly between runner-api and proto:

- `ReadOnly`
- `WorkspaceWrite`
- `DangerFullAccess`
- `Custom`

`RemoteToolRunner` sends typed requests directly and does not perform
capability gating before execution.

## Contracts

- `filesystem` is mandatory on every `ExecuteTool` request.
- There is no fallback decode path from any removed legacy field.
- There is no secondary filesystem policy projection on the wire.
- Typed filesystem transport is baseline behavior for all supported gRPC
  peers.

## Compatibility

This spec intentionally removes the old transport compatibility layer. Older
peers that depend on deleted fields are unsupported.

## Tests

Regression coverage should keep:

- typed filesystem enum round-trip
- remote runner execution of a typed filesystem request
- missing `filesystem` yielding `INVALID_ARGUMENT`
