# Inferred Contact Save UX

## Summary

The Contacts screen now reflects inference as a save operation. Running Infer
adds new inferred contacts directly to the saved contact list instead of
requiring a second Use/Create step.

## Design

- The desktop `inferContacts` API expects the daemon's refreshed
  `ContactsSnapshot` and optional `savedCount`.
- After inference completes, the screen applies the returned snapshot, selects
  the first newly saved contact, and reports how many contacts were inferred.
- The infer dialog keeps trace visibility and still displays legacy proposal
  cards when loaded from older stores, but successful new inference normally
  leaves no proposal cards.
- Partial legacy `contacts_infer` responses that contain proposals and
  candidates but no refreshed `contacts` list are merged into the existing
  snapshot so saved contacts stay visible and legacy proposal cards remain
  usable.
- Browser fake-daemon tests mirror the daemon by auto-saving inferred contacts
  and skipping contacts that overlap saved identities.
- Browser remote daemon handshakes accept the same shorthand query aliases used
  by launch URLs (`backend`, `token`, and `workspaceRoot`) as well as the older
  `pufferRemote*` names.

## Compatibility

Users who previously saved proposal cards manually can still edit or delete the
resulting contacts from the main list. Existing legacy proposal rows are not
broken; they can still be opened and used from the modal until the next
inference run clears stale proposals.
