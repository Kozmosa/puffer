# Desktop Auth Provider Aliases

## Scope

Desktop auth and agent creation share a provider vocabulary, but the UI may send
front-end canonical ids (`openai`, `anthropic`) while the local Tauri backend
stores and runs CLI-backed providers as `codex` and `claude`.

## Change

- API-key login now canonicalizes `openai`/`codex` to `codex` and
  `anthropic`/`claude` to `claude` before storing credentials.
- API-key login rejects unknown provider ids instead of writing unusable
  credential entries.
- Logout applies the same canonicalization before removing stored credentials,
  so either alias disconnects the active provider entry.

## Compatibility

The existing `puffer` provider id is unchanged. Model listing, session creation,
and default model selection already accept both UI aliases and backend ids, so
this keeps the auth store aligned with the rest of the desktop backend.
