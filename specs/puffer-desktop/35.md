# Real daemon WebSocket protocol

## Purpose

The desktop shell must use the real `puffer daemon` for normal Personal Agent
sessions instead of silently binding itself to the legacy in-process desktop
backend.

## Design

Tauri startup keeps the legacy backend available for transitional commands, but
the frontend's daemon client now asks Tauri to start the real local daemon before
building the shared WebSocket client. Browser previews still connect only when a
daemon URL is explicitly configured.

The TypeScript daemon client accepts both envelope formats during migration:
legacy desktop test fixtures still send `{ type: "response", ok, result }`, while
the real daemon sends `{ id, result | error }` and `{ event, payload }`.

## Logic

- WebSocket URLs include the daemon token as `?token=...`.
- Requests use string ids, which the real daemon requires.
- Response handling accepts real daemon errors as structured `{ code, message }`
  objects.
- `ensureLocalDaemonClient()` invokes `ensure_local_daemon` in Tauri and connects
  to the returned handshake.

## Contracts

The legacy in-process backend remains registered for command compatibility while
frontend callers migrate. Tests may opt into either fake protocol so regressions
in real-daemon framing are caught without losing older coverage.
