# Chat Composer Attachments

## Scope

Desktop chat now mirrors the agent-sdk composer attachment entry point: a `+`
button sits immediately before the model picker, opens a top-aligned dropdown,
and exposes `Add images and files`.

## UX Contract

- The composer owns transient attachment drafts per session.
- The file picker accepts images, PDFs, text, common code files, office
  documents, and archives.
- A message can be sent with text, attachments, or both.
- Image drafts render as 64px thumbnails backed by object URLs.
- Non-image drafts render as fixed-size file cards with filename and extension.
- Each draft has an `x` remove button.
- Drafts are capped at 10 files, 20 MiB per file.
- Dragging files over the composer shows a copy drop overlay.
- Object URLs are revoked when drafts are removed, accepted, or the component is
  destroyed.

## Data Contract

The frontend sends attachment metadata as `AgentTurnOptions.attachments`:

```ts
{
  id: string;
  name: string;
  mimeType: string;
  size: number;
  extension: string;
  kind: "image" | "file";
}
```

The current daemon path ignores unknown turn fields, so this is intentionally a
lightweight contract rather than a binary upload transport. Until daemon
staging consumes these refs directly, desktop also folds attachment summary
lines into the turn text so attachment-only messages are not persisted or sent
as empty prompts.

## Long-Term Upload Path

Do not send file bytes or base64 through the WebSocket turn RPC. The stable path
is a daemon-owned staging API:

1. Desktop stages selected files into a session-scoped upload directory.
2. The daemon returns stable `AttachmentRef` records with id, name, MIME, size,
   content hash, and staged path.
3. `run_agent_turn` receives refs, not bytes.
4. Provider adapters convert refs at the boundary:
   - images become provider-native image content parts when supported;
   - text/code/PDF files become bounded extracted text blocks or tool-readable
     staged paths;
   - unsupported file kinds remain visible as explicit file refs.
5. Transcript storage records user content as structured text/image/file parts,
   not just a single text field.

This keeps the UI responsive, avoids WebSocket memory spikes, and leaves one
binary lifecycle owner: the daemon/session store.
