# Desktop Browser Resize Failure Scoping

## Context

Browser resize events are asynchronous. The resize observer sent a
`browser_resize` request using the currently active backend session id, but its
failure callback wrote directly to the pane error state. If the user resized or
reflowed Browser in one agent and switched agents before the failure returned,
the new agent could display the old resize error.

## Change

- Capture the active Browser command target before sending resize requests from
  the resize observer.
- Reuse the Browser target-liveness check before applying resize failures.
- Suppress resize failures when the pane has been disposed, the generation
  changed, or the active backend browser session no longer matches.

## Compatibility

Current-tab resize failures still surface in the Browser pane. The change only
drops stale resize failures from an older active tab or older agent session.

## Verification

- Added a Playwright regression that delays an Alpha agent resize failure,
  switches to Beta before it resolves, and asserts Beta remains connected
  without rendering Alpha's stale error.
