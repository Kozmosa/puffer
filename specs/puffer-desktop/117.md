# Clock-Skewed Submitted Prompt Reconciliation

## Scope

The desktop chat view renders a local submitted user row immediately while the
daemon starts a turn. Same-session reloads then replace that optimistic row
with persisted transcript rows when the daemon reports generated titles,
renames, or turn completion.

## Change

- Persisted transcript rows that have already reused a transient local id now
  satisfy the pending-row reconciliation check even if their timestamps are
  outside the local five-minute clock window.
- This prevents a remote daemon or skewed host clock from rendering both the
  optimistic submitted prompt and the daemon's persisted echo of the same
  prompt.
- A Playwright regression test submits a prompt, reloads a transcript whose
  persisted `user_message` timestamp is ten minutes behind the renderer, and
  asserts only one user row remains visible.

## Compatibility

The timestamp window is still used for body-only matches so intentional later
repeats of the same prompt remain visible. The id match applies only after the
normal transient-id reuse step has identified a persisted row as the backend
copy of a local pending row.
