# Workspace Session Fallback History

## Scope

When a new session is created, the daemon can return the session id before the
next grouped-history refresh contains that session. The detail view and sidebar
already keep the fallback session visible, and the workspace board must do the
same after the user navigates back.

## Change

- The app derives workspace groups from grouped daemon sessions plus the
  currently selected session when that session is missing from the grouped
  history.
- If the fallback session belongs to an existing project path, it is inserted
  into that project with the normal recent-session ordering.
- If the fallback session's path is missing entirely, the workspace board gets
  a synthetic one-session group for that path.

## Compatibility

Persisted grouped sessions remain authoritative once the daemon catches up. The
fallback is only rendered when the selected session is absent from the current
grouped-history snapshot, so duplicate rows are avoided.
