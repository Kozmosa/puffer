# Stop empty MCP settings reload loop

## Purpose

The MCP settings pane is a normal Personal Agent setup flow. A workspace with no
configured MCP servers must render the empty state once instead of continuously
polling the daemon.

## Design

The pane now tracks whether the MCP list request has completed independently
from the returned server count. An empty result is a valid loaded state, not a
signal to refetch.

## Logic

- `mcpLoaded` starts false and flips true after the first list attempt,
  including error responses.
- The lazy-load effect checks `!mcpLoaded` instead of `mcpServers.length === 0`.
- Successful add-server responses mark the list loaded because they replace the
  local list from the daemon response.
- The fake daemon accepts an explicit MCP server fixture so tests can exercise
  empty workspaces.

## Contracts

No daemon RPC shape changes. The desktop UI continues to call
`list_mcp_servers` and `add_mcp_server`; it just stops conflating "empty" with
"not loaded."
