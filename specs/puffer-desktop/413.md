# Started Session Routing Lock

## Scope

Desktop chat routing preferences must not override the provider/model of a session that already has persisted conversation history.

## Behavior

- Session-scoped saved routing remains available for empty sessions before the first persisted turn.
- Once daemon metadata reports `eventCount > 0`, the composer uses the session's own `providerId` and `modelId`.
- Provider switching stays unavailable for started sessions.
- Stale `localStorage` routing records cannot silently send a later turn to a different provider.

## Compatibility

This keeps the existing empty-session provider switch flow intact while matching the locked routing semantics already implied by the disabled provider switch UI after conversation start.
