# Desktop Browser New Tab Failure Scoping

## Context

Browser new-tab creation is asynchronous. The success path already ignored stale
responses after a session switch, but the failure path still wrote the error
directly to pane state. If a user opened a new tab in one agent and immediately
switched to another agent before the daemon returned an error, the second
agent's Browser pane could show the first agent's failure.

## Change

- Capture the root session id used for the new-tab request.
- Use that captured session id in the daemon request.
- Ignore delayed new-tab failures when the Browser pane has been disposed, the
  session generation changed, the tab state version changed, or the active root
  session no longer matches the request.

## Compatibility

Same-session new-tab failures still surface normally so the user can see and
retry real Browser daemon errors. Only stale failures from an older agent or
older tab state are suppressed.

## Verification

- Added a Playwright regression that delays an Alpha agent new-tab failure,
  switches to Beta before the failure resolves, and asserts Beta remains
  connected without rendering Alpha's Browser error.
