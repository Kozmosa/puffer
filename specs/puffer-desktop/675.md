# Connected Image Provider Settings

## Summary

The Image generation settings modal should display only providers that are both
connected and executable for image generation. It should not show every
connected provider, and it should not apply OpenAI-specific controls to
non-OpenAI providers.

This is a breaking UI/settings contract update. Desktop should consume the
daemon's parameterized media capabilities as the source of truth.

## Modal Contract

On modal open, desktop calls `list_media_capabilities` with `kind=image`.

The provider dropdown is derived from returned `status=available` capabilities.
The model dropdown is derived from capabilities for the selected provider. A
connected provider that lacks executable image capability does not appear in the
dropdown.

The UI should render parameter controls from capability metadata:

- `select` becomes a select control;
- `number` becomes a numeric input with min/max/step when supplied;
- `boolean` becomes a checkbox or toggle;
- `string` becomes a text input only when the adapter exposes a bounded,
  user-editable string parameter.

The modal must not special-case provider ids for request parameters. Provider
labels may still use display names from the capability DTO.

## Settings Contract

Saved image settings should be:

```text
provider_id
model_id
adapter
parameters
```

The previous fixed image settings fields (`size`, `quality`, and
`output_format`) are no longer the desktop source of truth. Existing saved
values may be discarded and replaced by the selected capability defaults.

Save is enabled only when:

- settings are loaded;
- the selected provider/model/adapter exists in current capabilities;
- all required parameters have valid values.

If a saved selection becomes unavailable, the modal shows the stale selection as
invalid and asks the user to choose an available model. It must not silently
fall back and save a different provider.

## Loading And Refresh

Capability loading remains lazy. The app should not resolve image capabilities
while loading settings snapshots, workspace lists, or session lists.

The modal may expose a refresh action that reissues capability resolution. It
should preserve the user's unsaved local form state when the selected
capability remains available.

Provider-level unavailable reasons can be shown in a compact diagnostic area
only when they help explain why a connected provider is absent. The default
dropdown remains limited to available capabilities.

## Error Handling

Provider discovery and local probe failures are non-fatal to the modal if at
least one other image capability is available. The modal shows a no-capability
state only when the daemon returns no available image capabilities.

Generation errors continue to come from `generate_media`; the settings modal
does not attempt to test-generate images.

## Tests

Desktop coverage should include:

- OpenAI controls render from capability metadata rather than hard-coded fields;
- BytePlus/Zhipu/xAI-style capabilities render different parameter names;
- OpenRouter/Vercel router models appear only when daemon returns available
  capabilities;
- connected image-input-only providers do not appear;
- stale saved selections remain visible as invalid;
- refresh preserves compatible unsaved parameter values;
- no-capability and partial-failure states are accessible and keyboard usable.
