# Native CEF Robustness

## Design

Puffer Desktop keeps CEF as the preferred Browser renderer, with screencast as
the compatibility renderer when native CEF is unavailable. The macOS CEF bridge
must make Tauri's live `NSApplication` class satisfy CEF's `CefAppProtocol`
before `CefInitialize`, because Chromium selects its AppKit message pump based
on the app class' Objective-C protocol conformance.

## Architecture

- `cef_host_mac.mm` installs `CefAppProtocol` on the actual `NSApp` class and
  swizzles that class' `sendEvent:` implementation, not just the
  `NSApplication` superclass.
- The existing `isHandlingSendEvent` accessors remain process-wide state,
  matching CEF's requirement that `CefScopedSendingEvent` can mark AppKit event
  dispatch while Chromium handles nested run loops.
- `scripts/release.sh` applies a generated Chromium checkout shim when a CEF
  branch still includes `content/public/common/child_process_id.h` but the
  target Chromium checkout has moved `ChildProcessId` to
  `content/public/browser/child_process_id.h`.

## Contracts

- macOS CEF initialization must happen after the AppKit application exists and
  before Chromium creates its browser-process message pump.
- Release builds may patch generated Chromium checkout state, but the repo
  working tree only records the repeatable release script and this spec.
- The Linux compatibility shim is only generated when the old header path is
  missing and the new browser header exists.

## Compatibility

This preserves the CEF-default desktop behavior and the screencast fallback
surface. The Linux CEF release build remains compatible with Chromium 146/CEF
7680 checkouts where CEF patch files have not fully caught up to the header
move.
