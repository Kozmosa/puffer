# 293. Cancel Turn Reports Failure for Unknown Turn IDs

## Summary

The daemon's `cancel_turn` handler now returns `{"ok": false, "error": "turn not found"}`
when the requested turn ID does not exist in the active turn registry, instead of
unconditionally returning `{"ok": true}`.

## Behavior

- If the turn ID is found, the cancel token is triggered, pending permissions are
  denied, pending questions are drained, and `{"ok": true}` is returned.
- If the turn ID is not found (already completed, or never existed), the response
  is `{"ok": false, "error": "turn not found"}`.
- The frontend `cancelCurrentTurn` handler already tolerates RPC errors; this
  change gives it accurate feedback about whether the cancel had any effect.

## Compatibility

Clients that only check for RPC-level errors (thrown exceptions) are unaffected.
Clients that inspect `response.ok` can now distinguish a successful cancel from
a no-op.

## Tests

`chat-session-ui.spec.ts` verifies that cancelling a turn whose completion event
already arrived does not leave the UI in a stuck "Cancel requested" state.
