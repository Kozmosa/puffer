# puffer-desktop Browser Rapid Tab Creation

## Purpose

Rapid repeated new-tab clicks must allocate distinct browser tab ids even when
daemon create-tab responses are still pending.

## Design

The browser pane reserves the next tab id before awaiting the daemon. This keeps
multiple in-flight create requests from sharing the same local tab id.

## Logic

- `addTab` reads the current `nextTabNumber`, builds the requested tab id, and
  immediately advances the counter.
- Accepted daemon responses still recompute `nextTabNumber` from the resulting
  tab list, preserving recovery from daemon-provided ids.
- The existing tab-state version guard still ignores delayed create responses
  after newer authoritative tab snapshots.

## Contracts

Each user click on the new-tab button must map to a unique daemon tab id. Fast
clicks may complete in any order, but they must not overwrite each other.
