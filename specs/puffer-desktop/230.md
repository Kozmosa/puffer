# 230. Workspace Hides Child Sessions

## Summary

The desktop workspace board now uses a root-session view of daemon folder
groups. Sessions with `parentSessionId` are treated as child/subagent sessions
and are not promoted into workspace history, project rows, or project detail
boards.

## Behavior

- The shell derives workspace-facing groups from daemon groups by dropping
  child sessions.
- Group session counts are recalculated after filtering.
- Groups that contain only child sessions are hidden from the top-level
  workspace board.
- Direct session-detail APIs can still load a child session, but workspace
  navigation no longer presents it as a standalone user session.

## Compatibility

No daemon protocol changes are required. The UI uses the existing
`parentSessionId` field that already travels with session list items.

## Tests

`workspace-ui.spec.ts` covers a project with both parent and child sessions and
asserts that history, project rows, and project detail only show the parent.
