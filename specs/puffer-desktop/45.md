# Reject empty API key login attempts

The desktop provider login form now treats empty and whitespace-only API keys
as invalid before any daemon request is sent. The API-key connect button stays
disabled until the field contains a non-empty value after trimming, and Enter
uses the same validation path.

The legacy desktop backend also trims API-key login input and rejects empty
provider ids or empty keys. This keeps the fallback Tauri bridge from writing
blank credentials that would make a provider look connected while the first
real turn fails later.

Compatibility contract:

- OAuth and credential-import login flows are unchanged.
- Non-empty API keys are trimmed before being sent or stored.
- Empty API-key submissions never call `login_with_api_key` from the desktop
  UI.
- Legacy backend callers receive a validation error instead of a refreshed
  settings snapshot for empty API-key input.

Regression coverage lives in `settings-ui.spec.ts` for the user-facing form
and in `backend.rs` unit tests for the legacy validation helper.
