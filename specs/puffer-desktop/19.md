# puffer-desktop Browser Tab Mutation Versioning

## Purpose

Pending browser tab creation responses must not reinsert stale tabs after the
daemon has reported a newer tab-list state.

## Design

The browser pane tracks a local tab-state version. Daemon tab-list updates,
optimistic tab closes, and accepted new-tab responses advance the version.
`New tab` captures the version before awaiting the daemon and ignores the
response if another tab mutation arrived first.

## Logic

- Accepted tab-list events increment the tab-state version.
- Optimistic tab close increments the tab-state version.
- New-tab responses only update local UI when their captured version still
  matches the current version.
- Stale new-tab responses after a tab clear leave the UI in the no-pages state.

## Contracts

The desktop UI treats the daemon tab list as authoritative once it arrives.
Delayed create-tab responses must not override a newer daemon tab snapshot.
