# Browser Active Tab Normalization

## Purpose

The managed Browser pane must not trust a daemon `activeTabId` that is absent
from the accompanying tab list.

## Design

Tab-list state now validates the daemon active id against the received tabs
before storing it. If the active id is missing, the pane falls back to a
connected tab or the first listed tab.

## Logic

- Incoming tab info is converted before active id selection.
- The daemon active id is accepted only when it matches one converted tab id.
- Invalid active ids cannot become the basis for `browser_resize`,
  `browser_open`, or later input/navigation session ids.

## Contracts

The UI may receive inconsistent tab snapshots during daemon restarts or races.
The Browser pane must keep its visible active tab and backend session id aligned
to an actual tab from the latest snapshot.

The Playwright harness emits an invalid active id with a valid tab and asserts
that no request is sent to the missing backend session.
