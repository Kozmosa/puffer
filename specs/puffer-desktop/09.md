# puffer-desktop Browser Tab State Races

## Purpose

The managed Browser pane must keep tab state isolated when daemon responses
arrive after the user has switched to another tab.

## Design

Browser state application now carries the tab id captured by the async
connection attempt. Responses still update their owning tab record, but they
only update the visible URL, title, loading, error, and status fields if that
tab is still active.

## Logic

- Browser frame, state, and devtools subscriptions capture the tab id selected
  when the connection starts.
- `applyState` accepts a tab id and writes the response to that tab.
- Stale open or resize fallback responses do not overwrite the active tab's
  visible URL/status after a fast tab switch.
- Connection errors from stale attempts update only their owning tab record.

## Contracts

Daemon Browser requests may resolve out of order. The UI contract is that
out-of-order responses cannot mutate another tab's visible state.

The Playwright harness includes a delayed `browser_open` response regression so
future Browser refactors preserve this isolation.
