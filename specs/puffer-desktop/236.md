# 236. Provider Connection And Pending Model State

## Summary

The desktop UI now makes provider connection and model-loading state visible in
the two places where users make routing decisions: Settings -> Providers and the
chat composer model picker.

## Design

- Provider cards derive their connection state from the settings auth snapshot
  with provider-id alias equivalence. Cards show a Connected or Not connected
  badge, include the active credential kind/email when available, and relabel
  repeat actions as reconnect/update rather than first-time connect.
- The composer model picker switches its visible provider immediately when the
  user picks another provider. While the first model catalog request is in
  flight, the trigger and results pane name the pending provider and the
  composer disables send until a model is selected.
- Chat-visible engineer labels use the selected provider before falling back to
  the persisted session provider, so OpenRouter/custom provider selection is not
  masked by stale session metadata.
- The Settings default-model control shows an explicit per-provider loading
  note while daemon model discovery is pending.

## Compatibility

This is a frontend-only state and copy update. The daemon request contract for
`list_provider_models`, `run_agent_turn`, and `update_config` is unchanged, and
existing generation guards still drop stale model-list responses.

## Tests

Playwright coverage now checks that delayed OpenRouter model loading updates the
picker and chat label before the catalog resolves, keeps Send disabled while no
model is selected, and that Providers cards distinguish connected and
disconnected providers.
