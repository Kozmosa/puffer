# Files Watch Handshake Race

## Context

The Files pane subscribes to `workspace:fs:changed` before awaiting the
`fs_watch` response. Native watchers can emit a first change event before the
RPC response arrives, especially when an agent writes into an already-open
workspace immediately after the pane mounts.

The pane previously learned the daemon watch id only from the response. That
left `currentWatchId` unset while the response was pending, so the first real
event for the new watch was ignored as unrelated and the open editor could stay
stale until a later filesystem event.

## Change

- Generate a client-side filesystem watch id before sending `fs_watch`.
- Pass that id to both daemon implementations through the existing optional
  `watchId` RPC field.
- Treat the generated id as current while the request is pending, so early
  `workspace:fs:changed` events can refresh cached directories and the active
  file.
- Keep stale cleanup guarded: if the pane is rerooted or destroyed before the
  response arrives, the returned watch is still unwatched.

## Compatibility

Daemons that return the requested id preserve the new early-event behavior.
Explicit `fs_unwatch` semantics remain unchanged and continue to be
best-effort on stale ids.

## Verification

- Added a Playwright regression that delays the `fs_watch` response, emits a
  matching `workspace:fs:changed` event while the response is pending, and
  expects the active editor to refresh from the daemon's updated file content.
