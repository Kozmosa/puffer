# Files Save Session Guard

## Context

The Files pane awaited `write_file` without binding the response to the
session/root that launched it. When two agents shared the same cwd and active
path, a late save failure from the previous session could show as a visible
error in the newly selected session. A late success could also refresh file
cache state after the user had moved on.

## Change

- Capture the active root, session id, generation, and draft content when a
  save starts.
- Apply save success, failure, and `saving` cleanup only while that captured
  context still matches the visible Files pane.
- Preserve same-session saves and tab switches by keeping the existing active
  path check scoped inside the session/root guard.

## Compatibility

The daemon `write_file` protocol is unchanged. The guard only suppresses stale
client-side state mutation after the visible session changes.

## Verification

- Added a Playwright regression that delays a save failure in one session,
  switches to another session with the same cwd/path, and asserts the stale
  failure does not surface in the new Files pane.
