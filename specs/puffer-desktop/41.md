# Preserve permission edits during lazy loading

## Purpose

The Permissions settings pane lazy-loads tool policies from the daemon. A late
`list_permissions` response must not overwrite rows the user has already
started editing.

## Design

Permissions loading now has explicit loading state plus a request generation
guard. Initial loading disables row edits, and late responses only rebuild the
editable row copy when the user has no unsaved local changes.

## Logic

- Opening the Permissions pane starts at most one pending load.
- While the initial load is pending, Add rule and Save are disabled and the pane
  shows `Loading permissions...`.
- Stale responses are ignored by generation id.
- Fresh responses update the snapshot/path, but preserve dirty local rows.
- Saving still sends the editable row set to `save_permissions`.

## Contracts

No daemon API contract changes. This is a desktop UI state-management fix that
protects unsaved settings edits from asynchronous daemon responses.
