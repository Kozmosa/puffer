# puffer-desktop Update 668: Composer Media Settings

## Summary

The desktop composer should expose media configuration from the existing `+`
menu without folding media model selection into the chat `ModelPicker`.

The `+` dropdown has three entries:

- `Add images and files`
- `Image settings`
- `Video settings`

`Image settings` and `Video settings` open separate modals. Each modal shows
only connected providers with a verified capability for that media kind.

The first implementation is a settings and explicit-trigger surface. It should
not infer media generation from arbitrary chat text.

## Research Basis

OpenAI separates direct image generation from conversational image generation:
the Images API selects an image model directly, while the Responses API lets a
mainline chat model call an `image_generation` tool. See
https://developers.openai.com/api/docs/guides/image-generation.

Video and many third-party media providers are long-running job systems. fal
recommends queue-based async inference with status polling, streaming, webhooks,
and cancellation. Replicate models every run as a prediction and warns that
generated file URLs can expire. See
https://fal.ai/docs/documentation/model-apis/inference/queue,
https://replicate.com/docs/topics/predictions/lifecycle, and
https://replicate.com/docs/topics/predictions/output-files.

## UX Contract

- The composer menu remains a lightweight action menu. It does not start media
  generation by itself.
- `Add images and files` keeps the existing file picker behavior.
- `Image settings` opens an image-only modal with provider, model, operation,
  size, quality, output format, and partial-preview defaults when supported.
- `Video settings` opens a video-only modal with provider, model, operation,
  aspect ratio, duration, and resolution defaults when supported.
- The chat `ModelPicker` continues to select the reasoning/chat route only.
- Media modals may use a dedicated `MediaModelPicker`, but they must filter by
  media capability rather than general chat model availability.
- The first implementation persists global media defaults. Session-scoped media
  overrides are deferred unless they can reuse an existing session preference
  mechanism without adding a new storage contract.
- Media generation starts from explicit `/image` and `/video` prefixes, or from
  a future dedicated media command surface. Plain chat input remains chat input.

## Settings Contract

The desktop settings snapshot should expose a media settings object:

```ts
media: {
  image: {
    providerId: string | null;
    modelId: string | null;
    defaults: {
      operation?: "generate" | "edit";
      size?: string;
      quality?: string;
      format?: string;
      partialImages?: number;
    };
  };
  video: {
    providerId: string | null;
    modelId: string | null;
    defaults: {
      operation?: "text_to_video" | "image_to_video" | "video_to_video";
      aspectRatio?: string;
      durationSec?: number;
      resolution?: string;
    };
  };
}
```

No backward-compatible migration is required for this change. The long-term
schema should be direct and typed rather than derived from `defaultProvider` or
`defaultModel`.

The initial desktop config update should write the whole media branch in one
patch. Individual controls should update local modal state until the user saves.

## Capability Contract

Desktop should request media capability records from the daemon instead of
hand-maintaining model lists in Svelte:

```ts
type MediaCapabilityInfo = {
  providerId: string;
  providerDisplayName: string;
  modelId: string;
  modelDisplayName: string;
  kind: "image" | "video";
  operations: string[];
  asyncJob: boolean;
  streaming: boolean;
  parameters: Record<string, string[] | number[] | boolean>;
  status: "available" | "unavailable" | "unknown";
  reason?: string;
  source: "provider_api" | "adapter" | "builtin_hint" | "agent_inferred";
  checkedAtMs: number;
};
```

Only `status: "available"` capabilities are selectable by default. Unknown or
agent-inferred capabilities can be shown behind a refresh/test affordance, but
they cannot be executed until the backend validates them.

The UI should not interpret model names to guess capability. It only consumes
daemon capability records.

## Loading And Empty States

- Capability loading is lazy: open the relevant modal before calling the daemon.
- Loads are scoped by media kind and connected provider set.
- Results may be cached for a short TTL and refreshed manually.
- If no connected provider has a verified image or video capability, the modal
  shows a connect-provider path and a refresh-capabilities action.
- If a previously selected media model becomes unavailable, the modal keeps the
  stale value visible as invalid and asks the user to choose a valid model.
- Video settings must work even before any video adapter exists: the modal
  should show a clear no-capability state rather than hiding the entry.

## Performance And Stability

- Do not fan out discovery requests from composer render. Discovery starts only
  on modal open, manual refresh, or settings load where the data is already
  cached.
- Modal loads must be cancellable or generation-scoped so stale results cannot
  overwrite newer selections.
- Media settings writes should be a single config patch, not multiple partial
  writes from each control.
- Generated image or video bytes must not be stored in Svelte state beyond the
  preview lifecycle. Persistent media display should reference artifacts created
  by the backend.
- Preview UI should render artifact thumbnails or local URLs supplied by the
  daemon. It should not construct long-lived data URLs.

## Boundaries

- No automatic prompt classification is required for the first implementation.
  Use explicit `/image` and `/video` triggers first.
- No full workflow DSL, marketplace, or generic media plugin surface belongs in
  this change.
- No object-storage abstraction is required for desktop-first behavior; local
  artifact URLs or paths are enough.
- The existing chat provider picker should not grow media-specific parameters.
- Do not add session media persistence in the first pass unless the existing
  session preference path can be reused with no new backend schema.

## Brainstorming Review

The review tightened the design in four places:

- Keep media settings separate from chat routing, but start with global media
  defaults to avoid adding a second session-routing system.
- Keep capability selection automatic, but make the daemon the authority. The
  desktop does not derive capability from provider names or model strings.
- Keep the video settings entry visible even when no video provider is ready, so
  the UX contract is stable while backend support grows.
- Keep triggers explicit. Natural-language media intent classification and a
  composer media mode are useful later, but they are not required for the first
  stable path.

## Execution Plan

1. Extend desktop config DTOs, fake daemon state, and settings patch handling
   with the `media.image` and `media.video` branches.
2. Add a daemon request for `list_media_capabilities` filtered by media kind and
   connected providers. Return empty available lists without treating that as an
   error.
3. Add the two composer menu entries and modal shells. The modals should load
   capabilities lazily on open and save only on explicit user action.
4. Wire image settings to available image capabilities and invalid stale
   selections. Add the same path for video settings, including the empty state.
5. Add explicit `/image` and `/video` submit parsing only after backend media
   job endpoints exist. Until then, settings are selectable but do not imply
   execution.
6. Add Playwright and fake daemon coverage for menu behavior, modal capability
   filtering, empty states, invalid stale selections, and saved config patches.

## Verification

- Composer menu tests should assert all three menu entries and that the existing
  file picker path still works.
- Modal tests should cover connected-provider filtering, no-capability empty
  states, stale selection invalidation, and saved global defaults.
- Fake daemon tests should include image-only, video-only, both-capable, and
  no-capability providers.
- Accessibility checks should verify menu roles, modal focus trapping, labels,
  and keyboard dismissal.
