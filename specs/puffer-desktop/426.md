# Idle Reconnect Pending Start Recovery

## Context

If `run_agent_turn` reaches the daemon but the WebSocket closes before the
turn id response returns, the UI enters a recoverable pending-start state. When
the backend reconnects and reloads the session as idle with no persisted turn,
the composer could remain stuck in a disabled Stop state because the optimistic
submitted prompt was still missing from the transcript.

## Change

- Idle session reload now detects an orphaned pending start with no turn id and
  no live assistant/tool/prompt activity.
- That orphaned runtime state is cleared while preserving the visible
  optimistic user message.
- The regression confirms the user can send another prompt after reconnecting
  to an idle transcript.

## Compatibility

Active daemon sessions are unchanged. If the backend reports the session as
running or awaiting, the UI continues to keep the pending state instead of
unblocking the composer early.
