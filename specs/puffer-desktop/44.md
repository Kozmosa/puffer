# Reset Browser pane state on agent changes

The desktop Browser pane now treats `sessionId` changes as a full browser
context switch. It reloads saved tabs for the new session, drops active tab
subscriptions from the previous session, clears cursor/pointer state, and
re-subscribes to the new `browser:<session>:tabs` channel.

Async Browser operations carry a session generation guard. If a delayed tab
sync, browser-open, or resize fallback returns after the user has switched to a
different agent, the old response is ignored instead of updating tabs with the
same `tab-1` id in the new session.

Compatibility contract:

- Existing per-session saved browser tabs are still restored from localStorage.
- Empty daemon tab lists still fall back to the local saved tab until the daemon
  creates or reports a real page.
- Late tab-list events from the previous agent cannot alter the currently
  visible Browser pane.
- New navigation after switching agents targets the new session's browser
  backend id.

Regression coverage lives in `desktop-ui.spec.ts` and switches from one agent's
Browser pane to another while emitting a late tab-list event from the old agent.
