# Native CEF Startup Robustness

## Scope

This update hardens the macOS native CEF renderer used by the desktop Browser
pane.

## Design

- The Tauri setup hook initializes native CEF before the AppKit event loop
  starts.
- CEF commands require that setup-time initialization has already succeeded.
  They no longer perform late lazy initialization from a Browser-pane command.
- Runtime discovery prefers explicit environment paths, then bundled app
  runtime paths, then the compile-time development CEF root.
- Status reports CEF as available only after successful setup-time
  initialization. If initialization fails or never runs, the renderer remains on
  the screencast fallback path.

## Compatibility

- Existing development builds still use `~/chromium_tintin` when no bundled CEF
  runtime is present.
- Packaged apps can place `Chromium Embedded Framework.framework` and
  `cefsimple Helper.app` under `Contents/Frameworks`.
- Development builds link the CEF framework into both `target/Frameworks` and
  `target/<profile>/Frameworks` so app binaries and test binaries can load the
  same runtime.
- The Browser settings toggle remains unchanged: CEF is the default renderer and
  screencast is the compatibility fallback.

## Failure Behavior

CEF initialization errors are logged during desktop startup and exposed through
`browser_cef_native_status`. The app continues running and the Browser pane uses
screencast instead of attempting late native CEF startup.
