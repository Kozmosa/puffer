# CEF Daemon Target Prewarm

The macOS desktop app pre-creates hidden native CEF browser targets so the
local daemon can attach browser sessions to real embedded CEF browsers.

- `PUFFER_CEF_PREWARM_TARGETS` controls the number of hidden targets and
  defaults to `3`.
- Hidden targets are created with native CEF APIs and then hidden; they still
  publish normal DevTools page targets on the CEF remote debugging port.
- The daemon CEF path now claims these prewarmed page targets from `/json/list`
  instead of calling `/json/new`.
- Only blank `about:blank` page targets are eligible for daemon reuse, so
  smoke-test pages or visible desktop browser pages cannot be claimed by an
  agent browser session.
- When a daemon browser session closes, its native CEF target is returned to
  the reusable pool instead of being closed.
- If the reusable pool is exhausted, the daemon returns a CEF capacity error
  instead of starting an external Chrome fallback.
- When the desktop launcher advertises `PUFFER_CEF_REMOTE_DEBUGGING_PORT`, CEF
  startup or discovery failures are surfaced as CEF errors rather than silently
  falling back to an external Chrome process.
- Recycled native CEF targets are explicitly navigated back to `about:blank`
  when a new browser tab does not request another URL, so a newly opened tab
  cannot show a previously closed tab's page.

Compatibility: this avoids CEF DevTools-created targets, which are not stable
in the embedded macOS host. Multi-session browser work stays inside the native
CEF process as long as the prewarmed target pool has capacity.
