# Provider Reconnect Recovery Coverage

## Summary

Desktop tests now cover the recovery path where an existing session is blocked
because its provider is disconnected, the user reconnects that provider in
Settings, and the same session becomes sendable without reopening the app.

## Design

`FakeDaemon` now updates its auth snapshot when `login_with_api_key` or
`login_with_oauth` succeeds. `chat-session-ui.spec.ts` uses that behavior to
drive a cross-page flow from a disabled Claude session to Settings provider
login and back to the active composer.

## Contracts

- A provider login response must refresh the auth snapshot used by the active
  conversation view.
- Reconnecting a session's provider must remove the reconnect hint and enable
  the composer.
- The next turn must use the recovered session id, provider id, and model id.

## Verification

- `npm run test:desktop-ui -- tests/chat-session-ui.spec.ts -g "reconnecting a provider re-enables an existing blocked session"`
