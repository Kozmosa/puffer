# Refresh settings after daemon workspace changes

Switching the desktop daemon changes the authoritative workspace root, auth
store, permissions file, MCP manifests, and default model config. The desktop
shell now treats a daemon/workspace change as a full settings-source change
instead of only refreshing the session board.

Compatibility contract:

- The app refreshes the settings snapshot after a workspace switch.
- Daemon event listeners are rebound to the new shared daemon client, so future
  session and pin events come from the active workspace.
- Settings displays the daemon identity owned by the shell state, not a stale
  singleton read captured by the pane.
- Provider model caches, permission rows, and MCP server listings are reset
  when the settings source changes, then lazily reloaded for the active pane.
- In-flight session subscriptions and live turn state are cleared on daemon
  switch so events from the old workspace cannot update the new view.

Regression coverage lives in `settings-ui.spec.ts` and verifies that a refreshed
workspace root updates provider defaults and forces permissions to reload from
the new workspace path.
