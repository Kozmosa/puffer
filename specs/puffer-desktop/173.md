# Desktop Browser Input Failure Scoping

## Context

Browser canvas input is asynchronous. Mouse, wheel, keyboard, and paste events
sent `browser_input` requests using the active backend session id, but their
failure callbacks wrote directly to the pane error state. If the user interacted
with Browser in one agent and switched agents before a delayed failure returned,
the new agent could show the old input error.

## Change

- Reuse the scoped Browser command target helper for Browser input events.
- Capture the active backend browser session id and session generation before
  sending `browser_input`.
- Suppress input failures when the pane has been disposed, the generation
  changed, or the active backend browser session no longer matches.

## Compatibility

Current-tab input failures still surface in the Browser pane. The change only
drops stale failures from an older active tab or older agent session.

## Verification

- Added a Playwright regression that delays an Alpha agent mouse-input failure,
  switches to Beta before it resolves, and asserts Beta remains connected
  without rendering Alpha's stale error.
