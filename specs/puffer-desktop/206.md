# Alias Default Model Routing

## Context

Desktop New agent creation sends a provider id and often relies on the daemon
to select the model from the saved default config. Provider ids can be stored
or submitted through user-facing aliases such as `codex`/`openai` and
`claude`/`anthropic`.

The daemon canonicalized the requested provider before comparing it with the
stored default provider. If the config still used an alias, the comparison
failed and the session silently fell back to the provider's first advertised
model instead of the configured default model.

## Change

- Compare create-session default-provider ownership through the same desktop
  provider canonicalization used for request routing.
- Preserve alias-prefixed default models such as `codex/gpt-5.4` and
  `claude/claude-opus-4-6` when creating a new desktop session.
- Apply the same alias-aware comparison to the Tauri fallback backend.

## Compatibility

Explicit `modelId` values continue to win. Unknown or provider-mismatched
models still fail validation instead of being accepted silently.

## Verification

- Added daemon regression coverage for `codex` -> `openai` and
  `claude` -> `anthropic` default model selection.
- Added Tauri backend alias-matching coverage for the fallback path.
