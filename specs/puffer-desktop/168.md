# Desktop Chat File Link Line Targets

## Context

Chat markdown already parses local file links such as `/tmp/puffer/src/main.rs:30`
into a path and one-based line number, but the desktop detail view only forwarded
the path into the Files pane. Clicking a line-qualified link opened the file at
whatever cursor position the editor already had, often the end of the restored
tab, which made code-review and agent-instruction links feel broken.

## Change

- Represent chat file-link opens as request objects containing `path`, `line`,
  and a monotonically increasing request id.
- Forward the request through `AgentDetailContent` into `FilesPane` as path,
  line, and request id props.
- Treat the request id, not just the path, as the duplicate guard so repeated
  clicks and same-file different-line clicks still run.
- After the target file is opened or loaded, focus the editor, set the textarea
  selection to the requested one-based line start, and keep the overlay gutter
  and highlight layer scrolled with the textarea.

## Compatibility

Unqualified file links keep the existing behavior and only open the file. Normal
file-tree clicks and restored tabs do not move focus because they do not carry a
line target. The link parser remains responsible for stripping the `:line`
suffix before the filesystem RPC runs.

## Verification

- Added a Playwright regression that clicks a chat link to
  `/tmp/puffer/src/main.rs:30` and asserts the Files editor selection lands at
  the start of line 30.
