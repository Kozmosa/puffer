# PDF and Legacy Word Preview Hardening

## Design

PDF previews must not depend on a WebView-compatible PDF worker. The Files pane
now renders PDF pages with pdf.js on the main UI thread and keeps extracted text
visible below the rendered pages, so a blank or slow canvas path still leaves
readable content on screen.

Legacy Word-family previews use macOS `textutil` for both plain text and HTML
when available. The front-end sanitizes the converted HTML before rendering it,
preserving basic document structure such as headings, bold text, and tables
without allowing scriptable markup.

## Contracts

- File reads cap returned content at the existing preview hard limit instead of
  rejecting large files solely because their total size exceeds that limit.
- Large reads are truncated in the response and still report the real file size.
- PDF, `.doc`, `.dot`, and `.rtf` preview paths continue to request the rich
  preview byte cap from the daemon.

## Compatibility

Existing Markdown, CSV, OOXML, and raw text editing behavior is unchanged.
Older Word files without native `textutil` output still fall back to the
existing binary string extraction path.
