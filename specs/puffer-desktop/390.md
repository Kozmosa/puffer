# 390. Desktop Pin Race Recovery

Desktop pinning now treats daemon pin-change events as authoritative
acknowledgements. When a `desktop:pins:changed` event confirms the in-flight
state and there is no queued opposite intent, the matching pin busy key is
cleared before the RPC response returns.

If the user toggles the same agent or workspace pin while the previous request
is still in flight, the latest desired state is queued instead of being silently
dropped. The UI applies the queued state optimistically and sends the follow-up
request after the current request settles if the daemon state still differs.

Reconnect and daemon-scope resets clear in-flight and queued pin state so stale
busy flags cannot leak across daemon identities.
