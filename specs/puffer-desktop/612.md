# Browser native CEF renderer

## Summary

The desktop Browser surface now prefers native CEF embedding when the local
Tintin CEF build is present. New installations default to `cef`, settings can
switch back to `screencast`, and the screencast path remains the compatibility
fallback.

## Design

- `DesktopPreferences.browserRenderer` stores `cef` or `screencast`.
- `App.svelte` defaults the preference to `cef` and clamps older localStorage
  values to the supported renderer set.
- `Settings.svelte` exposes the renderer control and passes it into Browser
  panes, including connector browser-auth setup.
- `BrowserPane.svelte` asks both the daemon and the Tauri process for renderer
  status. If native CEF is available, it opens, resizes, navigates, reloads, and
  closes browser tabs through Tauri commands instead of subscribing to
  screencast frames. Recoverable native CEF command failures demote the active
  pane to the screencast path for the session, leaving the settings-level
  renderer preference intact for later retries.
- `src-tauri/src/cef_host.rs` exposes the CEF Tauri commands and discovers the
  local runtime, helper executable, remote debugging port, and profile cache
  root.
- `src-tauri/src/cef_host_mac.mm` initializes CEF in the Tauri process on
  macOS on the first native Browser command, explicitly loads the CEF framework
  before using the dynamic wrapper, points CEF at the framework resource
  directory, swizzles the `NSApplication` send-event hook expected by CEF, hosts
  child `NSView` containers over the Browser viewport, and pumps CEF's external
  message loop using CEF's `OnScheduleMessagePumpWork` callback with a
  reentrancy guard. The bridge never calls `CefDoMessageLoopWork` from fixed
  timers or command handlers.
- `src-tauri/build.rs` detects the local Tintin raw CEF build under
  `~/chromium_tintin/src/out/Release_GN_arm64`, compiles the generated CEF C++
  wrapper sources with the local Apple toolchain, and links the local
  `Chromium Embedded Framework.framework`. It also creates a dev-time
  `target/Frameworks/Chromium Embedded Framework.framework` symlink so the
  framework's macOS install name resolves for the `target/debug/corbina`
  executable, plus helper-app symlinks for the ANGLE/Vulkan dylibs that CEF
  subprocesses load relative to the helper executable in raw GN builds.
- The legacy `SettingsView.svelte` mock surface and mock preferences include
  the new field so type checks stay aligned.

## Compatibility

The existing daemon Browser RPCs, tab persistence, CDP screencast frames,
input forwarding, DevTools event streams, and recording fallback remain
available when the preference is `screencast` or when native CEF is unavailable.
Native CEF input is handled by the embedded Chromium view itself, so the Svelte
pane suppresses synthetic CDP input forwarding while CEF is active.
