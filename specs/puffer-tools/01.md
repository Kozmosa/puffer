# Model-visible tool filtering for internal Browser

## Purpose

Keep Browser executable as an internal runtime tool while removing it from the
model-facing tool list sent to providers.

## Design

`ToolRegistry` now separates all executable registered tools from model-visible
tool definitions. `tools()` and `tool()` continue to expose every registered
runtime entry. `definitions()` filters internal Browser definitions whose
handler is `runtime:browser`.

`PUFFER_NO_BROWSER=1` still disables Browser registration entirely. With the
browser daemon enabled, Browser can be looked up and executed locally, but it is
not returned by `definitions()`.

## Logic

Provider adapters build OpenAI and Anthropic tool payloads from
`ToolRegistry::definitions()`, so the Browser schema, description, and input
params are not included in LLM API calls. Runtime code that needs direct Browser
dispatch can still resolve the registered tool by id or alias.

Registry tests were split into a dedicated visibility test module so touched
Rust files remain under the 1000-line repository limit.

## Contract

`definition("Browser")` is a registry lookup contract and may return a runtime
definition when Browser is enabled. `definitions()` is the provider-advertising
contract and must not return internal Browser.

## Verification

- Registry tests assert disabled Browser is not registered.
- Registry tests assert enabled Browser is executable but absent from
  model-visible definitions.
- Provider tool-visibility tests assert OpenAI and Anthropic payloads exclude
  Browser.
