# Internal tool registry

## Purpose

Separate provider-callable tools from CLI-only internal tools without relying on
visibility filters or command parsing.

## Design

`ToolRegistry` keeps two independent maps:

- `tools` and `aliases` for normal model-visible/provider-callable tools.
- `internal_tools` and `internal_aliases` for first-party CLI-only tools.

`from_resources` registers `LoadedResources::tools` into the normal maps and
`LoadedResources::internal_tools` into the internal maps. `definitions()`,
`tool()`, `definition()`, `execute()`, and `execute_json()` only use the normal
maps. `internal_tools()`, `internal_tool()`, and `internal_definition()` are the
explicit internal lookup surface.

## Contract

Provider adapters must build request tool payloads only from `definitions()`.
An internal tool definition must not be returned by `definition()` and must not
be executable through normal model tool-call dispatch.

Browser is registered from `resources/internal_tools/browser.yaml`; it is
available through `internal_definition("Browser")` and absent from
`definition("Browser")`.

## Verification

Registry visibility tests load both bundled tool directories. They assert that
normal bundled tools are provider-callable, internal bundled tools are
registerable through the internal surface, and Browser is not present in normal
definitions.
