# puffer-genskill-eval Update 05

## Summary

Fixes Ladybird corpus target-test execution so replay outcomes are based on
real targeted tests instead of CTest patterns that do not match imported web
test files.

## Design And Architecture

The replay harness keeps deriving target filters from files in each corpus
entry's `tests/` directory, but command generation now splits filters by test
runner family:

- `Tests/LibWeb/...` HTML filters run through Ladybird's `test-web` target.
- C++ unit test files run through `Meta/ladybird.py run` using the executable
  target name derived from the source filename stem.

Unsupported filters fail explicitly with status `127` instead of silently
passing.

## Logic

LibWeb test paths are converted from repository paths such as
`Tests/LibWeb/Text/input/foo.html` to `test-web` filter paths such as
`Text/input/foo.html`. Multiple web filters are passed to one `test-web`
invocation with repeated `-f` arguments.

For C++ files such as `Tests/LibRegex/TestRegex.cpp`, the harness runs the
unit target `TestRegex`, matching Ladybird's `ladybird_test()` executable
naming. This avoids `Meta/ladybird.py test`, which builds the default target
set before CTest and is too broad for one focused corpus check.

## Contracts

- A replay must not be marked pass or wrong-fix from a target-test command that
  reports `No tests were found`.
- Corpus test path derivation remains deterministic and sorted.
- The command generator must keep failing unknown filter shapes visibly.

## Verification

- `cargo test -p puffer-genskill-eval` passes 23 tests.
