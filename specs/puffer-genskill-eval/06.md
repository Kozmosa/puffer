# puffer-genskill-eval Update 06

## Summary

Stabilizes Ladybird replay target tests for minimized corpus entries by adding
explicit per-entry test filters and reducing Ladybird build parallelism during
target-test execution.

## Design And Architecture

Corpus entries may include an optional `test_filters.txt` file next to
`meta.json`. When present, replay uses those filters instead of deriving
filters from every file under `tests/`. This lets a minimized corpus carry one
large upstream C++ test file while running only the regression cases introduced
by the PR.

The filter format is line-oriented:

- blank lines and lines beginning with `#` are ignored
- `Tests/LibWeb/...html` entries run through `test-web`
- `Tests/.../Target.cpp` runs the full C++ unit target
- `Tests/.../Target.cpp:case_glob` runs only matching cases inside that unit
  target

## Logic

Ladybird `Meta/ladybird.py run` invocations now pass `--jobs 1` for target-test
builds. This avoids compiler OOM failures on Docker hosts with limited memory,
especially for LibWeb generated binding translation units.

The two LibGfx image decoder corpus entries use explicit C++ case filters so
the replay checks only the PR regression cases and no longer requires the full
upstream image fixture tree.

## Contracts

- `test_filters.txt` is deterministic: filters are sorted and deduplicated
  after comments and blank lines are removed.
- Existing path-derived filter behavior remains the fallback for corpus entries
  without `test_filters.txt`.
- Low build parallelism is part of replay correctness because OOM during target
  test setup is an environment failure, not an agent outcome.

## Verification

- `cargo test -p puffer-genskill-eval` covers explicit filters, C++ case
  filters, and low-parallelism command generation.
