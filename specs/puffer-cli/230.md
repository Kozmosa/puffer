# Monitor Rule RPCs

## Design

The daemon exposes `task_monitor_rule_add` / `monitor_rule_add` and
`task_monitor_rule_delete` / `monitor_rule_delete` for monitor-owned include
and exclude rules. The UI submits intent:

- `connection_slug`
- `mode`: `exclude` or `include`
- `keywords`
- `case_insensitive`

The daemon validates the monitor binding, compiles keywords as literal text
contains filters, and persists the resulting `FilterSpec`. Connector payload
fields are intentionally ignored by this RPC; rule behavior is keyword-only
across the selected monitor connection.

## Persistence

- `exclude` appends a unique rule to `ignore_filters`.
- `include` writes to binding `filter`; multiple include rules are represented
  as `any`.
- Deletion accepts the exact displayed rule JSON, removes matching rules, and
  simplifies empty or single-item include compounds.

## Snapshot Contract

`workflow_list` includes monitor-facing `include_filter`, `include_filters`,
and `ignore_filters`. It does not expose connector-aware field metadata for rule
editing.

## Tests

Daemon tests cover exclude persistence, include persistence, delete behavior,
per-monitor isolation, and the absence of `scope_options` in monitor snapshots.
