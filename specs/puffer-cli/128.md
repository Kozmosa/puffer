# Gmail Browser Draft Persistence

## Summary

Make Gmail Browser draft actions persist drafts instead of only opening a
compose URL.

## Design

- `draft_reply` and `draft_forward` require at least one recipient, subject, or
  body field.
- Standalone draft actions open the Gmail compose URL, wait for Gmail readiness,
  verify or populate expected body and subject fields, then wait for Gmail
  autosave.
- Thread actions wait for a real Gmail thread view, not only generic Gmail
  readiness, so they do not run against a stale inbox while navigation is still
  rendering.
- `draft_reply` with a thread id opens the target thread, clicks Gmail's reply
  control when the reply editor is not already visible, writes the reply body,
  and waits for Gmail autosave in the thread.
- Standalone compose drafts navigate to Drafts and confirm a matching draft row
  is visible before returning success. Failures include the Gmail script status
  or reason so callers do not report a saved draft when Gmail never exposes or
  persists a draft.

## Compatibility

Existing standalone compose URL generation and send behavior are unchanged.
Threaded `draft_reply` now behaves as a real Gmail reply draft instead of a new
message compose. Draft actions now have stronger completion semantics, so
callers should expect failures for empty draft inputs that would not create a
visible Gmail draft.
