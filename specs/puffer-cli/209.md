# Contact Store Sanitization

## Summary

Contact store loading now normalizes saved contacts and inferred proposals so
legacy or malformed `contacts.json` rows do not make persisted inference results
disappear after reopening the desktop UI.

## Design

- `load_store` sanitizes saved contacts, inferred proposals, avatars, and
  contact ids before returning the store snapshot.
- Saved contacts with overlapping normalized contact ids merge into one record
  while preserving the first non-empty name, description, and avatar.
- `save_proposals` sanitizes new proposal data before returning it to callers
  and before writing it to `.puffer/runtime/contacts.json`.
- Proposal pruning still removes suggestions that overlap normalized saved
  contact ids.

## Compatibility

Existing stores remain JSON-compatible. Stores without a `proposals` field still
load with an empty proposal list, while stores with legacy id formatting are
read through the normalized contact id rules used by save and inference paths.
