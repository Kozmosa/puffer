# Contact Daemon APIs

- The daemon exposes `contacts_list`, `contacts_search`, `contacts_context`,
  `contacts_save`, `contacts_delete`, and `contacts_infer` JSON-RPC methods.
- Saved contacts are user-authored records stored under workspace runtime config and
  contain a label, description, optional avatar, and one or more normalized connector
  contact ids.
- Telegram candidates are ranked from diagnostics NDJSON using message entropy,
  reply behavior, recency, read/reply latency, personal chat days, and acceleration.
  Group senders are ignored unless the user replies or mentions them.
- Google/email/calendar candidates are inferred from subscription history payloads;
  context returns recent high-signal emails or events, while Telegram context returns
  high-entropy deduped messages.
- Command-backed connector-native contact lists are merged into daemon candidates when
  a connection exposes the standard contact operation. Connector-native context is
  queried before falling back to locally ranked history/diagnostic context.
- Contact inference asks OpenAI for JSON-only `CreateContact` tool calls when an
  OpenAI key is available and falls back to deterministic one-candidate proposals.
- Monitor creation accepts `contact_ids`. Missing contact ids preserve an existing
  monitor scope, explicit empty lists mean all contacts, and workflow snapshots expose
  the saved scope.
