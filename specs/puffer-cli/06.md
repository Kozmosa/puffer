# Desktop session routing sidecar

## Purpose

Desktop-created agent sessions can carry an initial provider/model choice before
the first user turn. That routing must survive the immediate workspace refresh
without changing the core session metadata schema.

## Behavior

- `create_session` accepts `providerId` and optional `modelId`.
- When a provider is supplied, the daemon resolves a valid default model for
  that provider and stores the pair in `desktop_state.json`.
- `list_grouped_sessions` and `load_session_detail` attach the stored routing
  fields to their session DTOs.
- The `create_session` response echoes the resolved provider/model so the UI
  can open the new session immediately.

## Compatibility

Sessions created without routing fields behave as before. Core session metadata
remains unchanged; the routing sidecar is desktop-specific and optional.
