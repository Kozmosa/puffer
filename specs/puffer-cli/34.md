# Internal tools and permission command surface

## Purpose

Move third-party-style tools such as Browser away from model-advertised tool
calls and expose them through a CLI surface that agents can invoke with Bash.
This keeps provider tool payloads small while preserving local runtime
integration.

## Design

`internal_tools.rs` introduces the `ThreePpTool` trait. Each implementation
declares:

- a stable tool id
- shell aliases to install or print
- the skill name that teaches agents how to use the alias
- command dispatch behind `puffer internal-tool <tool> ...`

The initial implementation is `BrowserThreePpTool`, backed by the existing
browser CLI command parser and runtime handler.

## Logic

`puffer internal-tool aliases` prints shell alias declarations such as:

`alias browser='puffer internal-tool browser'`

`puffer internal-tool browser ...` dispatches to the Browser internal command.
The CLI keeps Browser available as a local runtime capability without requiring
Browser to appear in Anthropic or OpenAI tool definitions.

Auto-mode command-surface output now reports ACL defaults instead of sandbox
state. The default summary describes cwd read/write, preapproved shell command
commands, and project ACL rules. Yolo help text says that it bypasses permission
prompts.

## Compatibility

`puffer browser ...` remains available as a human CLI command. Agent-facing
Browser work uses the `browser` alias or the explicit
`puffer internal-tool browser ...` form, and those internal-tool invocations
require the runtime permission callback.

## Verification

- `cargo check -p puffer-cli`
- Browser permission tests cover the internal-tool callback route.
