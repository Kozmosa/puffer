# Persistent Monitor Triage Sessions

## Summary

Background monitor triage now reuses one task-agent session per connection and
model instead of creating a fresh context for every connector message.

## Design

- `ProcessWorkflowRunner` stores triage `AppState` instances in memory behind a
  `monitor-triage:<connection>:<model>` key.
- The runner implements `triage_agent_batch` and renders one prompt turn
  containing either a single trigger or a JSON array of same-connection
  triggers.
- Reused triage sessions set `prompt_cache_key_override` to the session key so
  stable monitor instructions and prior context can benefit from provider input
  caching.
- The normal single-trigger `triage_agent` method delegates to the batch path
  with one trigger, preserving existing action contracts.
- Ignore-analysis agents still run without tools and without the persistent
  triage session because they are advisory, not task-creation turns.

## Compatibility

The persisted task/session store format is unchanged. Persistent triage sessions
are process-local and are recreated after daemon restart, while connector cursor
ack semantics remain owned by `puffer-subscriptions`.
