# Contact Inference Trace Events

## Design

`contacts_infer` accepts an optional `trace_id`/`traceId` parameter. When set,
the daemon publishes replayable events on `contacts:infer:<trace_id>:event` so a
desktop client can show the prompt and in-flight work while the blocking RPC is
still running.

## Architecture

The contacts RPC remains responsible for candidate collection and persistence.
OpenAI request building and response parsing live in `daemon_contacts_infer`.
Trace event emission lives in `daemon_contacts_trace` and is a no-op when no
trace id is supplied.

Inference candidate collection uses local/cached connector context only. It does
not execute live connector contact commands, because those commands can block on
external services and leave the UI stuck on the `CollectContacts` running state.
It also keeps only three text snippets per contact and does not clone payloads;
the OpenAI prompt consumes text snippets, not full connector payload objects.
History-backed contact candidates are read through a bounded newest-run visitor,
so inference no longer clones and sorts the entire workflow history file before
emitting the `CollectContacts` result.

## Contract

Trace payloads are conversation-style objects:

- `message`: `id`, `role`, `title`, `body`, `createdAtMs`
- `tool`: `id`, `toolName`, `status`, `title`, `summary`, `input`, `output`,
  `createdAtMs`

Tool events reuse the same `id` for running/completed state transitions. Prompt
events include the system prompt and a bounded JSON candidate prompt preview.

## Compatibility

Existing `contacts_infer` callers can omit `trace_id` and receive the same final
response shape. Failed OpenAI inference still falls back to heuristic proposals,
now with trace messages that explain the fallback.
