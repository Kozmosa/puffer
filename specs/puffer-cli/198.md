# History Telegram Candidate Avatars

## Summary

History-derived contact candidates now reuse cached Telegram peer avatars in
the same enrichment path that already prefers cached Telegram display names.

## Design

- `contacts_list`, `contacts_search`, and contact inference load Telegram peer
  avatar metadata alongside peer display names.
- Workflow history candidates copy the cached avatar for matching normalized
  Telegram contact ids when the candidate has no explicit avatar yet.
- The per-history-run merge logic is factored into a testable helper so the
  avatar path is covered without installing process-wide subscription manager
  state in unit tests.

## Compatibility

Saved contact data remains user-owned. This change only enriches response
payloads and inferred/history candidates from local Telegram peer cache data;
it does not rewrite saved contacts or infer avatars from message text.
