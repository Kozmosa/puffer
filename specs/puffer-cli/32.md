# Browser daemon current-tab context RPC

## Purpose

Add a daemon-side read-only Browser RPC that returns the current active tab
context for one browser root session so the dedicated Browser permission
evaluator can hydrate Browser actions that omit an explicit URL.

## Design

The Browser daemon already tracks per-root tab state in
`crates/puffer-cli/src/daemon_browser/tabs.rs` and refreshes live tab metadata
from connected page workers. This update adds a separate current-tab context
view layered on top of that registry instead of teaching permission callers to
reconstruct it from the broader tab list.

The new JSON-RPC method is `browser_current_tab`. It takes `sessionId` as the
root browser session id and returns a structured snapshot with:

- `status`
- `tabId`
- `url`
- `origin`
- `host`
- `port`
- `title`

`status` makes the edge cases explicit so callers do not need to infer meaning
from missing fields alone.

## Logic

The daemon resolves the current-tab snapshot from the Browser tab registry:

1. Look up the active tab for the supplied root browser session.
2. If there is a live backend worker for that tab, refresh `url`, `title`, and
   loading state from the worker cache before building the snapshot.
3. Return one of four statuses:
   - `no_active_tab` when the session has no active tab
   - `empty_url` when the active tab exists but its URL string is empty
   - `about_blank` when the active tab URL is `about:blank`
   - `available` when the daemon can return parsed URL context
4. For `available`, parse and return:
   - `origin` for `http` and `https`
   - lowercase `host`
   - explicit or default `port`

The RPC is read-only. It does not mutate focus, open new tabs, or synthesize a
tab when the session has none.

Permission-request events emitted by the daemon now forward the resulting
Browser context to clients as a structured Browser approval payload, while
Browser scope selection remains backend-only.

## Contracts

- `browser_current_tab` must return a structured result for any `sessionId`
  instead of failing on no-tab state.
- The result must always carry `tabId`, `url`, `origin`, `host`, `port`, and
  `title` keys, with `null` where context is unavailable.
- `about:blank` must be distinguishable from an empty URL.
- The RPC must be implemented in the daemon Browser `rpc/session/tabs/state`
  path and remain read-only.
- Daemon `permission-request` events for Browser tool calls and shell-derived
  Browser approvals must expose the same Browser payload fields:
  `source`, `actionSet`, `url`, `origin`, `host`, and `targetClass`.
- Desktop daemon-consumer code and the Tauri fallback event path must forward
  the same Browser payload shape unchanged so frontend consumers can render one
  consistent Browser approval context regardless of transport.
