# Release Artifact Repository Split

## Scope

The release Makefile and `scripts/release.sh` now treat source checkout,
application artifacts, CEF runtime artifacts, and Chromium Chrome artifacts as
separate repository concerns.

## Design

- `SOURCE_GITHUB_REPO` defaults to `berabuddies/puffer` and is used when the
  remote Linux builder clones the Puffer source tree.
- `RELEASE_GITHUB_REPO` defaults to `berabuddies/puffer` and receives desktop
  release artifacts.
- `RELEASE_TAG` defaults to `0.0.1-alpha` so Puffer's first app release is
  published under the product version tag rather than the CT runtime tag.
- `CEF_GITHUB_REPO` defaults to `berabuddies/ct` and receives CEF runtime
  artifacts. Runtime downloads for Tauri also use this repo by default.
- `CEF_RELEASE_TAG` defaults to `ct` so app builds resolve CEF from the CT
  runtime release unless explicitly overridden.
- `CHROME_GITHUB_REPO` defaults to `berabuddies/ct` and receives Chromium
  Chrome artifacts from the local `chromium_tintin` build.
- `CHROME_RELEASE_TAG` defaults to `CEF_RELEASE_TAG`, so CT Chrome and CEF
  assets normally publish under the same release tag.
- `GITHUB_REPO` remains a compatibility input for older release invocations,
  but the explicit variables are preferred.
- `build-release-chrome` packages `Chromium.app` on macOS as a signed and
  verified zip. On Linux it packages the local Chromium runtime directory as a
  tarball and checks the staged `chrome --version` before upload.
- TUI tarball publishing is opt-in through `UPLOAD_TUI_ARTIFACTS=1`; the
  default Puffer release contains the CEF-built desktop app artifacts.

## macOS Packaging

The macOS package mutates the Tauri `.app` after Tauri builds it by adding the
Puffer CLI, bundled resources, and the Chromium Embedded Framework runtime. The
final bundle is signed only after those mutations.

At runtime, the desktop app resolves the CEF helper from the selected runtime
root before falling back to compile-time paths. Packaged apps therefore launch
the helper from `.app/Contents/Frameworks` instead of a developer machine's
`~/chromium_tintin` checkout.

CEF helper apps now receive copied sidecar libraries and Vulkan metadata rather
than symlinks back to the Chromium build output. This keeps the CEF package and
the final `.app` self-contained on machines without `~/chromium_tintin`.

The app and Chrome zips are created with `ditto` so macOS code-signing metadata
is preserved, then extracted with `ditto` and verified with
`codesign --deep --strict` before upload. This catches broken release zips
before they are published.

## Compatibility

The default CEF release tag still follows `RELEASE_TAG` unless
`CEF_RELEASE_TAG` is provided. Users can override any repository variable from
the environment or Makefile invocation without changing targets.
