# Internal Browser Permission Preflight

## Purpose

Make `puffer internal-tool browser ...` ask for Browser permission through the
structured internal-tool callback before it touches the browser daemon.

## Design

The public `puffer browser ...` command remains a human-facing browser control
surface. The hidden `puffer internal-tool browser ...` surface now runs the
same command implementation with `internal_permission_required = true`.

Before each daemon-backed Browser action, the CLI builds the Browser-native
payload and sends it to `puffer_tools::internal_permissions` with
`tool_id = "browser"`.

## Logic

For public `puffer browser ...`:

- If a parent runtime exported a permission callback, the CLI preflights the
  structured Browser action.
- If no callback is present, the command proceeds as a direct human CLI action.

For hidden `puffer internal-tool browser ...`:

- A callback is required.
- `allow` proceeds to the daemon request.
- `deny` aborts before the daemon request with the denial reason.
- Missing callback aborts before the daemon request.

## Contract

The CLI does not parse shell command text. It asks permission only from the
structured Browser payload it is about to send to the daemon.

- `cargo check -p puffer-cli`
- Browser command tests continue to cover Browser argument mapping.
