# Host-Scoped Daemon Singleton

## Summary

The daemon now acquires a host/user scoped lock at CLI startup before loading daemon runtime state or starting subscriber children. The lock lives at `<user-config>/daemon.lock`, which is shared across workspaces for the same `PUFFER_HOME`/host user configuration.

## Design

- `puffer daemon` opens `<user-config>/daemon.lock` and calls `try_lock_with_pid` before the main process installs workflow, heartbeat, or subscription runtimes.
- If the lock is already held, startup fails before binding a socket, writing a handshake, or starting background subscribers.
- The error includes the owning PID when available and the current default `daemon.handshake` URL/workspace when that file can be parsed.
- The guard is passed into the daemon runtime and held for the full lifetime of the daemon process, then released automatically when the process exits.

## Compatibility

- This intentionally changes daemon scope from one-per-workspace behavior to one-per-host-config behavior.
- Existing tests and isolated workflows can still run parallel daemons by using distinct `PUFFER_HOME` directories.
- Subscriber processes remain child processes, but only the single daemon that owns the host lock may supervise them.
