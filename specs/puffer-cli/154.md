# CT browser runtime and extension verification

The managed screencast browser no longer falls back to stock desktop Chrome.
Daemon-owned browser roots resolve an explicit `PUFFER_CHROME`/`PUFFER_CT_CHROME`
override first, then cached or packaged `berabuddies/ct` Chromium assets, then a
local `~/chromium_tintin` checkout. If none are present, the daemon downloads
the current-platform `chromium-tintin-chrome-*` asset from `berabuddies/ct@ct`
into `~/.puffer/browser-runtimes/ct`.

CEF runtime status uses the same CT release model. It recognizes
`PUFFER_CEF_PATH`/`PUFFER_CEF_ROOT`/`CEF_PATH`, packaged CEF roots, cached
`puffer-cef-*` release assets, and local Tintin CEF builds. The daemon still
uses screencast unless the desktop native CEF bridge is active, but status and
fallback reasons now describe the CT release path instead of implying stock
Chrome is acceptable.

When browser extensions are requested, startup now verifies that Chromium/CEF
actually registered an extension runtime before returning a usable browser root.
Configured captcha solver seeds now fail if no matching extension target can be
found, because otherwise the UI can report a saved solver while the extension
never receives its API key.

Compatibility implications:

- System Google Chrome is no longer an implicit managed-browser fallback.
- `PUFFER_CT_REPO`, `PUFFER_CT_RELEASE_TAG`, and `PUFFER_BROWSER_RUNTIME_DIR`
  can redirect the CT runtime source/cache for development and release testing.
- CEF release assets use the existing `puffer-cef-<platform>-<arch>.tar.gz`
  naming contract.
- Extension-enabled sessions fail loudly on a CT/CEF build that does not honor
  unpacked extension loading.
