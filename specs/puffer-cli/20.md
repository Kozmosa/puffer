# Fresh Desktop Model Lists

## Problem

Desktop model pickers used the daemon's runtime snapshot after startup-style
provider discovery. That path can apply a warm cache or silently keep bundled
provider YAML entries when discovery is skipped or fails, leaving stale models
such as unavailable OpenAI ids visible in the GUI.

## Design

The `list_provider_models` RPC now builds provider runtime inputs without the
all-provider cached discovery pass. It then runs foreground discovery only for
the requested provider before serializing the model descriptors.

## Contracts

- Desktop provider aliases such as `claude` and `codex` still canonicalize to
  `anthropic` and `openai`.
- Providers without discovery config continue returning their bundled models.
- Discovery-enabled providers with credentials, or local base URLs, return the
  freshly discovered authoritative model set.
- Discovery errors for the requested provider propagate to the RPC caller
  instead of falling back to stale bundled entries.

## Compatibility

Normal daemon startup and agent turns keep their existing all-provider discovery
behavior. The stricter foreground fetch applies only to the explicit desktop
model-list RPC used by Settings and the chat model picker.
