# 28. Same-Session Turn Guard

## Summary

The daemon rejects a second `run_agent_turn` for a session while another turn is
still in flight for that same session. This prevents desktop duplicate sends,
reconnect races, or manual clients from forking chat context by starting two
provider calls from the same transcript snapshot.

## Design

- In-flight turn handles already retain the session UUID.
- `start_turn` checks the active turn map under the same mutex used for
  insertion before registering the new turn.
- A conflicting request fails fast with `turn-start-error`, including the
  existing turn id in the message for UI/debug visibility.
- Turns for different sessions can still run concurrently.

## Compatibility

The desktop UI still owns normal send-button disabling and cancel behavior. This
daemon guard is a backend correctness layer for races that bypass the ordinary
UI path. Clients already handle `run_agent_turn` startup failures through the
existing RPC error envelope.

## Tests

`daemon_turn_smoke` starts a delayed provider turn, attempts a second
same-session turn, verifies the second request is rejected, and confirms the
mock provider only received the first call.
