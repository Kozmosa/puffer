# Startup Resume Picker Does Not Create Blank Sessions

## Problem

Launching `puffer --resume <query>` with an ambiguous query opened the TUI
resume picker by first creating a brand-new session. If the user only wanted to
choose an existing conversation, or if the process could not enter the
interactive TUI, that empty session still appeared in history.

## Design

- Direct exact `--resume` matches still load the matched persisted session.
- Ambiguous startup resume requests now build the picker with a transient
  in-memory session id (`Uuid::nil`) instead of calling `create_session`.
- The picker still emits `/resume <id>` for the selected existing session, so
  choosing a session replaces the transient state with the persisted target
  before writing transcript events.

## Test Coverage

- `startup_resume_picker_does_not_create_blank_session` seeds two matching
  sessions, runs `puffer --resume dockyard` in non-interactive mode, and asserts
  the expected TUI error does not leave an extra `.session.json` file behind.
