# Telegram Auth Probe Retry

## Summary

Telegram subscriber-backed auth checks now retry transient negative auth probe
responses before marking a connection as degraded.

## Design

- `TelegramConnectionAuthChecker` still handles only `telegram`,
  `telegram-user`, and `telegram-login` connector templates with a running
  connection-owned subscriber.
- Transport failures, subscriber restarts, and command timeouts remain
  indeterminate and leave persisted connection state unchanged.
- A terminal `auth_ok` response with `ok: true` succeeds immediately.
- A terminal `login_error` or `auth_ok` with `ok: false` is retried twice with a
  short delay before returning an explicit failed-auth result.

## Compatibility

This keeps the existing connection lifecycle and monitor suppression behavior:
persisted connections are still marked `degraded` only when the auth checker
returns an explicit failed result. The change narrows Telegram false positives
caused by subscriber startup races or short-lived auth probe failures without
altering Gmail, GCal, Slack, Lark, or external connector auth checks.
