# puffer-cli Update 159: Native CEF NopeCHA Registration

## Summary

Native CEF and daemon-owned Chrome browser roots preconfigure the bundled
NopeCHA automation extension through a staged manifest copy when a solver key is
available.

## Contract

- Built-in CAPTCHA extension launch settings may stage a copied extension under
  `~/.puffer/browser-extension-stage/<solver>`.
- NopeCHA automation builds receive `manifest.nopecha.key`,
  `manifest.nopecha._base_api`, and `manifest.nopecha.enabled` before the
  extension is loaded.
- The checked-in bundled extension resources are never modified with user
  secrets.
- NopeCHA does not require a live MV3 service worker for daemon runtime storage
  seeding; other solvers still use their existing runtime seed path.
- The desktop native CEF host and daemon launch settings use the same staged
  extension path so daemon extension registration checks match the actual CEF
  profile.
- `PUFFER_CEF_PROFILE_DIR` advertises the native CEF profile directory to the
  daemon. The daemon uses it as the Preferences fallback when CEF omits a
  dormant MV3 service worker from target discovery.

## Compatibility

Existing custom extensions and 2Captcha behavior are unchanged. NopeCHA without
a configured key still loads the bundled automation build without static paid
key registration.

## Verification

Unit tests cover NopeCHA manifest staging without mutating the bundled source
manifest. Native CEF verification should confirm the CEF Preferences file lists
the staged extension path and `browser_open` no longer fails when the NopeCHA
service worker is dormant.
