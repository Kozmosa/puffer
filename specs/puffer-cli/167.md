# Browser Target Discovery via CDP

- Browser target discovery now prefers the browser-level DevTools websocket and
  `Target.getTargets` before falling back to DevTools HTTP `/json/list`.
- Native CEF DevTools discovery probes IPv6 loopback before IPv4 loopback so a
  stale crashed IPv4 listener cannot mask a healthy `[::1]` CEF instance.
- Browser-level target discovery drops its short-lived websocket probe without
  an explicit close handshake to avoid leaving CEF DevTools unresponsive.
- Page workers and reusable-target resets also drop CDP sockets without an
  explicit close handshake, and reusable CEF targets reset outside the root
  allocation mutex so one slow release cannot block future tab opens.
- Native CEF targets do not start daemon CDP screencasts or churn screencasts on
  resize. The native renderer owns frames; CDP remains available for state,
  input, and evaluation.
- Opening an empty native CEF tab leaves the prewarm target's slot-marker URL in
  DevTools instead of navigating it to plain `about:blank`, preserving recovery
  metadata.
- The desktop native CEF default prewarm pool is eight targets. The
  `PUFFER_CEF_PREWARM_TARGETS` override remains authoritative.
- Native CEF remains compatible with reusable prewarmed slots whose URLs encode
  `about:blank#puffer-cef-slot=<slot id>`, but a hung `/json/list` endpoint no
  longer prevents daemon attachment when the browser websocket is healthy.
- Managed Chrome keeps the existing HTTP target listing fallback for older or
  mocked DevTools endpoints that do not support browser-level target discovery.
- Page websocket URLs are derived from the browser websocket host and port plus
  the target id returned by `Target.getTargets`, preserving IPv6 bracket syntax
  and the original websocket scheme.
