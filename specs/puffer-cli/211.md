# Inferred Contact Persistence

## Summary

Contact inference now treats model `CreateContact` outputs as saved contacts
instead of leaving them only as transient proposals.

## Design

- `contacts_infer` still collects ranked connector candidates and asks the
  inference model for contact records.
- Returned proposals are normalized through the same contact ID rules used by
  manual saves.
- Inferred contacts are written to `.puffer/runtime/contacts.json` as
  `SavedContact` rows with generated local IDs.
- Existing saved contacts are never replaced by inference. If an inferred
  contact overlaps any saved `contact_ids`, it is skipped.
- Stale proposal rows are cleared when inference runs so reloads do not show
  obsolete suggestions.

## Compatibility

The RPC response now returns the refreshed contacts snapshot plus `savedCount`.
Desktop callers can keep reading `contacts`, `candidates`, and `proposals`;
`proposals` will normally be empty after successful inference. Legacy proposal
loading remains supported for older `contacts.json` files.
