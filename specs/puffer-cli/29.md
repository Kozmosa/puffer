# 29. Cancelled Turn Release

## Summary

Cancelling a daemon turn releases the session's in-flight guard as soon as the
cancel event is persisted and sent to clients. Users can start a follow-up
message immediately after the UI reports cancellation, even if the old provider
request is still unwinding in the background.

## Design

- `cancel_turn` still cancels the token, drains pending permission/question
  prompts, and records the cancellation transcript event.
- After a successful cancellation report, the turn handle is removed from the
  daemon's active turn map.
- The worker thread keeps its existing cancellation-report guard, so a delayed
  provider cancellation does not emit a second error event.
- Same-session concurrent starts remain rejected for genuinely active turns.

## Compatibility

Desktop clients already treat the immediate cancellation event as the point
where the composer should recover. Releasing the daemon guard at that same
point makes the backend match the UI contract without changing the RPC shape.

## Tests

`daemon_turn_smoke` now covers cancelling a delayed provider turn and starting a
new same-session turn before the first provider request returns.
