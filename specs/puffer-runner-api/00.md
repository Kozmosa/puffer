# puffer-runner-api Typed Filesystem Contract

## Purpose

Define the final runner-facing filesystem contract exposed by
`puffer-runner-api`.

## Design

`FilesystemExecutionPolicy` is the canonical filesystem DTO at the runner
boundary.

`ToolRequest.filesystem` is required on every tool execution request and is the
only filesystem policy carrier.

The supported filesystem sandbox modes are:

- `ReadOnly`
- `WorkspaceWrite`
- `DangerFullAccess`
- `Custom`

`RunnerCapabilities` does not carry filesystem-transport compatibility flags.
Typed filesystem transport is part of the baseline contract, not an optional
feature.

## Logic

Runner callers construct a typed `FilesystemExecutionPolicy` and pass it
through `ToolRequest.filesystem`.

Runner implementations preserve that typed value end-to-end. Missing
filesystem policy is invalid input, not a signal to rebuild permissions from
another field.

## Contracts

- `ToolRequest.filesystem` is mandatory.
- Filesystem policy must remain typed across runner boundaries.
- Callers and transports must not collapse filesystem policy into a coarse
  boolean.
- `Custom` is a first-class runner-api mode even if downstream execution
  bridges remain narrower.

## Compatibility

This spec intentionally drops the old bool-based compatibility semantics.
Peers that do not send the typed filesystem field are outside the supported
contract.
